# 基于 witty-diagnosis-agent 的 Linux vmcore 智能诊断实践

## 1. vmcore 深度诊断的业务挑战与传统方案局限

在 Linux 操作系统的规模化运维与 AT/ST 测试中，底层系统级故障（如 Crash、Panic、Hang 死）的排查一直是运维领域的深水区。面对系统生成的 vmcore 内存转储文件，传统的排查模式与早期的通用大模型方案均暴露出显著的局限性。

### 1.1 vmcore 深度排查的特有挑战与特点
- **软硬耦合度高，定位链路长**：操作系统向下纳管海量异构硬件，向上支撑复杂的微服务应用。故障发生时，通常需要资深内核专家熟练运用 `crash` 工具，在数十 GB 的十六进制内存数据和汇编指令中溯源。
- **跨域协同成本高，MTTR 难以压缩**：从堆栈解析到最终在数百万行 Linux 内核源码中锁定 Bug 触发点，往往需要拉通底层研发、驱动开发等多个团队。初步定界耗时常在小时级，根因定位甚至需耗费数天。

### 1.2 通用大模型（单体 Agent）的场景失效
当尝试将单体 Agent 引入 vmcore 诊断时，通用模型的缺陷在严苛的运维场景中被放大：
- **上下文溢出与注意力失焦**：动辄数百 MB 的 `crash` 命令执行日志输出，极易导致大模型上下文溢出，使其在海量内存与寄存器数据的噪音中丢失初始诊断目标。
- **场景感知缺失导致的误判**：针对系统 Hang 死但运维人员“人为触发 (sysrq)” Crash 以保留现场的场景，单体模型常被 Crash 这一表象误导，错误地去分析“系统为何 Crash”，从而偏离了“死锁或资源耗尽”的真正根因。
- **浅层归因与逻辑断层**：模型往往在发现“空指针异常”或“内存越界”等直接报错后便停止推演，缺乏向下挖掘硬件故障或驱动逻辑缺陷的深度反思能力。

## 2. 架构破局：witty-diagnosis-agent 协同诊断解决方案

针对单体 Agent 的局限，我们引入了 **witty-diagnosis-agent 协同诊断框架**，将复杂的 vmcore 诊断过程解耦为“规划、编排、执行、分析”四个标准化阶段，通过各司其职的专职 Agent 协同，实现了从“单步探索”向“系统化全局推演”的跨越。

### 2.1 诊断规划 (Fuxi)：前置环境隔离与故障场景定界
**解决输入不确定性，确保诊断方向收敛。**

在执行具体诊断命令前，**规划 Agent (伏羲)** 负责获取并澄清前置条件：
- **诊断环境的特殊性校验**：vmcore 的离线诊断对环境依赖极高。伏羲会强制核对远程调试机是否具备“诊断三要素”：匹配目标系统架构的 `crash` 调试工具、与原故障环境内核版本完全一致的带有调试信息的 `vmlinux` 文件，以及核心的 `vmcore` 转储文件。同时，还会验证是否挂载了对应的 Linux 内核及驱动的**源代码文件**，为后续的源码级定界铺平道路。
- **环境隔离校验**：严格区分执行诊断代码的“本地诊断容器”与存放上述 vmcore 文件的“远程调试机”，防止环境探测命令的错投。
- **故障现象的精准转译与定界**：面对用户诸如“系统复位起不来，卡在重启阶段”这类非技术语言描述，伏羲会主动将其转译为具体的技术线索，并强制判断当前场景属于“系统原生 Crash”（如网卡驱动 Bug 直接触发 Panic）还是“人为触发的 Crash 快照”（如系统 Hang 死，运维人员通过 sysrq 强行获取的现场）。若为后者，伏羲将在输出的《诊断分析计划》中，明确将后续排查重心转移至系统死锁、网络阻塞或资源耗尽方向，避免被 Crash 表象误导。

### 2.2 编排与执行 (Dayu & Kuafu)：基于故障模式的并发全景扫描
**克服长文本干扰，构建“现象 -> 模式 -> 证据”的闭环排查链。**

面对海量 vmcore 数据，**编排 Agent (大禹)** 负责接收伏羲传递的初步定界结论，并基于经典的“假设-验证”方法论，将具体的故障现象映射为特定的**故障模式 (Fault Model)**，随后将其转化为有向无环图 (DAG)，调度多个 **执行 Agent (夸父)** 并发工作：
- **基于 Ansible 的跨节点远程执行**：由于承载 vmcore 与 crash 工具的诊断环境通常部署在独立的远程调试节点上，夸父 Agent 在挂载底层专家技能（Skills）后，会通过 Ansible 协议将分析脚本安全、高效地投递至远端服务器执行，并实时捕获远端返回的探测结果，确保诊断逻辑与目标环境的完美解耦。
- **基于故障模式的策略分发**：不再进行盲目的全局搜索，而是根据识别出的故障模式精准分发远程诊断任务。例如，若判定为“原生 Crash 模式”，则重点调度夸父提取 Panic 堆栈与寄存器状态；若判定为“系统 Hang 死模式”，则调度多个夸父实例同步下发脚本，分别对 CPU 调度队列（runq）、内存分布（kmem）、锁依赖（死锁检测）等维度执行并发全景扫描。
- **关键线索提纯与高噪隔离**：真实的 vmcore 文件通常高达数 GB 甚至数十 GB，直接解析会产生海量且充满噪音的常规内存打印数据。在我们的框架中，各个执行 Agent 之间实现了严格的物理与上下文隔离，它们在完成各自维度的探测后，仅向上层（白泽）反馈提纯后的高价值线索（如明确的异常堆栈、死锁等待图）。这种“沙盒式”的隔离与提纯机制，不仅有效屏蔽了排查过程中的海量无效信息，更从根本上避免了单体模型在面对 GB 级转储文件时必然遭遇的上下文溢出与幻觉问题。

### 2.3 根因分析 (Baize)：“假设-验证”闭环与源码级定位
**打破模型逻辑盲区，构建完整的因果证据链。**

各维度的排查线索最终汇聚至 **分析 Agent (白泽)**。白泽通过引入 **“反思 (Reflection)”机制**，执行深度的逻辑推演：
- **多维证据交叉验证**：不满足于单一报错（如空指针），而是结合内存状态与并发上下文，反问“是否为底层硬件状态异常导致？”或“是否存在竞态条件？”。
- **本地源码级定界分析**：在通过异常堆栈锁定故障函数与触发逻辑后，白泽会直接联动用户在远程调试机挂载的**本地源代码文件**。它不仅能将异常指令流精准映射至具体的 Linux 内核或驱动源码片段，还能深入分析该代码片段的上下文逻辑，提取并剖析出导致报错的具体代码缺陷（如变量未初始化、锁未释放等），从而输出极具深度的源码级定界报告。

## 3. 核心支撑：诊断技能 (Skills) 的资产化与演进

witty-diagnosis-agent 框架的高效运转，依赖于底层高质量的 OS 专属专家技能 (Skills)。我们构建了从生成到调优的全生命周期管理闭环，将专家的隐性经验转化为系统的显性能力：

1. **经验标准化与环境一致性对齐**：依托 witty-skill-insight 项目，支持将存量的 Markdown 格式运维排查手册自动转化为标准化 Skill。为了确保模型能够快速且无歧义地启动分析，我们在 Skill 中固化了对诊断文件环境的一致性描述标准（如强制要求包含 `src` 源码目录、`vmcore` 核心转储文件以及配套的 `vmlinux` 文件），彻底消除了因目录结构混乱导致的 Agent 启动失败。
2. **植入基于分场景的专业诊断工作流**：我们摒弃了“一招鲜”的通用排查逻辑，将专家级的“分场景分析策略”固化到 Skill 中。当面对“系统故障触发 (场景 A)”时，Skill 会引导 Agent 启动标准 RCA 模式，从崩溃堆栈倒推代码漏洞；而面对“用户主动触发 (场景 B，如系统 Hang 死主动抓包)”时，Skill 会强制 Agent 忽略触发动作本身，转而启动“定点清除模式”，专注于寻找系统死锁、运行队列阻塞或 OOM 等亚健康状态的证据。
3. **强制构建时间线与强证据链 (Evidence Chain)**：为了防止大模型陷入浅层归因（如仅仅停留在空指针异常），Skill 强制要求 Agent 在得出结论前必须完成“构建时间线”与“五问法 (5 Whys)”分析。Agent 必须通过 `log`、`bt -t` 等命令重构崩溃前的时间线，并严格按照 `[现象] -> [直接原因] -> [技术机制] -> [设计缺陷] -> [根本原因]` 的逻辑层级提交分析报告，且每个层级都必须附带诸如寄存器状态、反汇编代码或结构体未初始化等确凿的“实锤”证据。

## 5. 总结与未来规划

针对开篇提出的 vmcore 深度诊断业务挑战与传统单体模型的局限，基于 witty-diagnosis-agent 协同的智能诊断框架给出了系统性的解决路径：

1. **破解场景感知缺失与误判**：通过“规划 Agent (Fuxi)”的**环境前置定界**，强制识别“原生 Crash”与“人为触发快照”的场景差异，从源头确保排查方向的准确性。
2. **克服上下文溢出与注意力失焦**：通过“编排与执行 Agent (Dayu & Kuafu)”的**并发全景扫描与高噪隔离**机制，将 GB 级的海量内存数据转化为精准的高价值线索（如异常堆栈、锁等待图），彻底规避了长文本干扰。
3. **打破逻辑断层与压缩 MTTR**：通过“分析 Agent (Baize)”的**本地源码级联动与强证据链推演**，改变了以往“见异常即结案”的浅层归因模式。实现了从模糊现象到确切底层源码缺陷的穿透，将原本需要跨团队协同、耗时数天的定位过程压缩至分钟级。

这不仅验证了多智能体在深水区运维场景中的巨大潜力，更标志着底层故障排查从“依赖个人经验”向“标准化系统能力”的跨越。

未来演进将聚焦于提升系统的全局自动化水平与泛化能力：
1. **构建覆盖全栈的故障模型矩阵**：基于 Linux 常见的故障模型（如 hardlockup, hungtask, memory_error 等），横向扩展 Agent 的领域知识，建立跨存储、网络、计算维度的联合诊断体系。
2. **全自动端到端（Autopilot）诊断巡航**：深化 **轩辕 (Xuanyuan) 总控 Agent** 的调度能力。针对标准化高频故障，实现从告警触发、环境快照抓取到 vmcore 解析的全自动流水线作业，实现从辅助诊断向“自动驾驶”运维的跨越。
