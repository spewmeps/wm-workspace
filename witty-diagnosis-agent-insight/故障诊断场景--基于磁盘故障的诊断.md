# 基于 witty-diagnosis-agent 的磁盘故障智能诊断实践

## 1. 磁盘与存储诊断的业务挑战与传统方案局限

在数据中心的大规模集群运维中，存储系统是支撑上层业务的基石。然而，磁盘与存储链路故障往往伴随着极高的复杂性与隐蔽性。面对动辄数百 MB、甚至包含海量噪音的多源日志包（iBMC、OS Messages、InfoCollect），传统的排查模式暴露出显著的效率瓶颈。

### 1.1 磁盘与存储排查的特有挑战与特点
- **全量日志中的大海捞针**：一线收集上来的通常是包含 CPU、内存、网络、磁盘等所有子系统状态的“全量日志包”（动辄数百 MB 甚至 GB 级）。在缺乏精准过滤手段的情况下，如何从这片数据的汪洋大海中，精准抽取并剥离出与磁盘、RAID 控制器或存储链路强相关的碎片化日志，本身就是一个极具挑战的难点。
- **多源日志割裂，关联分析困难**：磁盘故障的线索通常散落在不同维度的日志中：底层的 iBMC 记录了硬件事件，操作系统的 `dmesg` 和 `messages` 记录了 SCSI/ATA 错误，而 `infocollect` 则包含了 RAID 状态与 SMART 信息。运维人员需要手动跨越这些海量且结构不一的文件，耗时耗力。
- **时序对齐门槛高，因果倒置频发**：在多节点或多组件并发报错的场景下，日志时间戳往往存在偏差或格式不统一。人工很难精准重构故障发生的时间轴（T0），极易导致“因果倒置”（例如将软件超时误判为硬件故障，或将磁盘亚健康引发的 OOM 误判为内存泄漏）。
- **MTTR 难以压缩**：从发现 I/O 报错，到提取证据，再到确认是物理坏道、RAID 降级还是驱动链路问题，整个排查周期极其漫长，容错率极低。

### 1.2 通用大模型（单体 Agent）的场景失效
当尝试将单体 Agent 引入磁盘日志诊断时，由于缺乏领域约束，往往会遇到以下问题：
- **上下文溢出与注意力失焦**：直接将原始的 `messages` 或数十万行的 iBMC SEL 日志喂给模型，瞬间就会撑爆上下文，导致模型在海量常规心跳打印中迷失。
- **缺乏时序推理能力**：单体模型通常只能针对单点日志进行“字符串匹配式”解读，无法像人类专家一样进行跨日志的交叉时序对齐与因果推断。
- **浅层归因与缺乏实锤**：模型可能看到 `I/O error` 就草草断定是“磁盘坏了”，却无法进一步定位到具体的物理槽位（如“Slot 8 磁盘发生 Media Error”），导致给出的结论无法直接指导换盘或修复。

## 2. 突破海量日志分析瓶颈：witty-diagnosis-agent 协同诊断解决方案

针对传统排查的痛点与单体 Agent 的局限，我们引入了 **witty-diagnosis-agent 协同诊断框架**。将复杂的磁盘诊断解耦为“规划、编排、执行、分析”四个标准化阶段，通过多 Agent 协同与底层诊断脚本（Skills）的深度绑定，实现了从“海量日志检索”向“高维时序因果推演”的跨越。

### 2.1 诊断规划 (Fuxi)：日志环境识别与前置定界
**解决输入混乱，确保排查方向收敛。**

在启动具体诊断前，**规划 Agent (伏羲)** 负责识别用户提供的日志环境并进行前置澄清：
- **故障现象与时间的精准锁定**：在基于海量日志的分析流程中，同一台机器在不同时间段可能存在多个不同的报错（即历史噪音）。因此，伏羲会主动与用户澄清并要求提供相对准确的**故障时间段**与**具体的故障现象**。只有精准锁定了时间窗口，后续的检索才不会被历史遗留问题误导。
- **日志包完整性校验**：伏羲会扫描挂载的日志根目录，识别是否存在 `ibmc_logs`、`infocollect_logs` 或 `messages` 等核心目录。
- **排查目标的智能转译**：面对用户诸如“帮我看看这台机器昨晚为什么存储变慢了”这类模糊输入，伏羲会将其结合确认后的时间点，转化为具体的排查目标，并智能决策出需要挂载的底层诊断脚本集合。

### 2.2 编排与执行 (Dayu & Kuafu)：并发提纯与全局扫描
**克服长文本干扰，提取高价值时序证据。**

面对繁杂的多源日志，**编排 Agent (大禹)** 接收到伏羲的排查目标后，会调度多个 **执行 Agent (夸父)** 并发工作，执行针对性的提取策略：
- **基于故障模式的多维度分析**：大禹不再盲目让 Agent 去“看日志”，而是将故障现象映射为特定的故障模式，下发多维度的并发探测任务。例如，若怀疑是底层硬件故障，则指派夸父挂载 `diagnose_ibmc.py` 去分析带外传感器日志；若怀疑是链路丢包，则指派夸父挂载 `diagnose_messages.py` 去分析 OS 层的 SCSI 状态码。
- **基于时间窗的关键线索定点提取**：夸父 Agent 严格遵循伏羲锁定的“故障时间段”，结合故障模式的关键词，在海量日志中执行“定点清除”式的关键信息提取。它们只向上层反馈匹配时间窗且含有实质报错的日志片段（如 `Drive Fault` 或 `SCSI error`）。这种机制彻底抛弃了危险的“全局全量扫描”，从根本上避免了模型上下文溢出或不可控的幻觉问题。

### 2.3 根因分析 (Baize)：软硬结合的时序因果推演
**打破模型逻辑盲区，构建完整的因果证据链。**

各维度提纯后的异常线索汇聚至 **分析 Agent (白泽)**。白泽不再孤立地看待单条报错，而是内置了强大的软硬件结合时序推演机制：
- **构建软硬故障时间线 (T0)**：白泽会从海量碎片中锁定“故障零点 (T0)”（通常是最高优先级的硬件底层报错时间）。以此为锚点，将硬件层（iBMC SEL）、内核驱动层（dmesg）以及 OS 应用层（messages）的事件投影到统一的绝对时间轴上，清晰重现故障的演进链路（例如：`T0: iBMC 报 Slot 8 介质错误` -> `T0+2s: 内核报 sd 驱动 I/O error` -> `T0+5s: 系统报 EXT4-fs error`）。
- **软硬结合的交叉因果验证**：基于构建的时间线，白泽会运用专家规则进行因果反思。如果是“硬件先行”，则铁定为物理故障引发上层雪崩；如果是“软件先行而 iBMC 无记录”，则果断推翻磁盘损坏的猜测，将排查方向转向系统配置错误、软锁死 (Soft Lockup) 或 I/O 资源瓶颈。
- **贯穿全链路的细粒度定界**：最终的诊断结论必须是贯穿整个故障链路的闭环。白泽会联动存储拓扑与映射文件（如 `diskmap.txt`），将 OS 逻辑层的报错盘符（如 `/dev/sdm`）精准追溯到具体的物理设备（如 `RAID Controller 1, Slot 8`），最终输出包含时间线、证据链与修复操作（如物理换盘指令）的完整根因报告。

## 3. 核心支撑：诊断技能 (Skills) 的资产化与演进

witty-diagnosis-agent 框架的高效运转，依赖于底层高质量的存储诊断技能 (Skills)。我们将专家的排查范式转化为标准化脚本，赋能给 Agent：

1. **环境一致性对齐与标准化工具箱**：在 Skill 中固化了对典型日志目录结构（iBMC/OS/InfoCollect）的识别逻辑，并提供了统一的 Python 诊断脚本集。这确保了 Agent 能够快速适配不同厂商（Huawei, Inspur, H3C 等）的底层日志格式，无缝启动分析。
2. **全局概览脚本（`diagnose_summary.py`）提供全景图**：为避免大模型陷入盲目搜寻，Skill 强制 Agent 优先执行 `diagnose_summary.py`。该脚本能迅速输出两项核心数据：
   - **时间范围对齐**：输出 `[1. Metadata & Time Analysis]`，确认各组件日志（iBMC, OS）的时间跨度是否覆盖故障时间，指导后续处理时区差异。
   - **关键错误扫描**：输出 `[2. Detailed Diagnosis Summary]`，提供故障的初步定性。例如，若 iBMC 报出 `Drive Fault` 则倾向**硬件故障**；若 OS 报出 `SCSI error` 但 iBMC 无异常，则倾向**链路或驱动问题**；若 OS 报出 `Soft lockup`，则指向**系统/内核问题**。
   > **成效**：这一步直接给大模型提供了一个高度浓缩的“全景图”，不仅极大地降低了模型在海量日志中迷失方向的误判几率，还显著减少了无效试探带来的巨大 Token 消耗。
3. **植入四步专业诊断工作流**：摒弃了无序的搜索，将专家级的“快速定性 → 时序关联 → 分层深究 → 交叉验证”四步法固化到 Skill 中。强制 Agent 先使用概览模式扫描全局致命错误，再根据时间线进行深度下钻。
4. **强制构建时间线与强证据链**：要求 Agent 在提交最终报告时，必须清晰呈现“故障零点 T0”以及事件发生的时序矩阵。每一个结论层级（从现象到根本原因）都必须附带跨层日志的交叉验证证据（如 iBMC 硬件报错与 OS I/O 报错的时间吻合度），杜绝浅层归因。

## 5. 总结与未来规划

针对开篇提出的磁盘与存储诊断业务挑战，基于 witty-diagnosis-agent 协同的智能诊断框架交出了一份系统性的答卷。它完美应对了存储排查的四大特点与难点：

1. **解决“全量日志中的大海捞针”**：通过“编排与执行 Agent”的**沙盒式隔离与提纯机制**，彻底摒弃了全量扫描，精准从包含 CPU、内存、网络等全局信息的数百 MB 乃至 GB 级日志包中，剥离出与磁盘和存储链路强相关的异常片段。
2. **解决“多源日志割裂与关联困难”**：通过固化在 Skill 中的**环境一致性对齐与标准化工具箱**，Agent 能够跨越 iBMC、dmesg、messages 和 InfoCollect 的文件结构差异，自动完成异构日志的结构化解析与汇聚。
3. **解决“时序对齐门槛高与因果倒置”**：通过“分析 Agent (Baize)”强大的**软硬结合时序推演能力**，精准确立故障零点 (T0)，并将多源事件投影到统一的绝对时间轴上。基于“硬件先行”或“软件先行”的规则交叉验证，彻底杜绝了将软锁死误判为磁盘物理损坏的“因果倒置”悲剧。
4. **解决“MTTR 难以压缩与浅层归因”**：通过**贯穿全链路的细粒度定界**，Agent 不再给出“I/O 报错”这种模棱两可的结论，而是直接联动拓扑映射文件，输出“物理槽位 Slot 8 磁盘坏道”的实锤证据，将原本漫长的排查周期压缩至分钟级。

这不仅验证了多智能体在复杂时序因果推断场景中的巨大潜力，更标志着存储运维从“人工堆时”向“自动化专家系统”的跨越。

未来演进将聚焦于提升系统的预测能力与全局拓扑感知：
1. **构建存储亚健康预测模型**：基于 SMART 属性和低频软错误（如间歇性 SCSI 超时）的时序趋势分析，横向扩展 Agent 的领域知识，实现从“事后定界”向“故障孕育期预警”的跨越。
2. **闭环的自动化故障修复（自愈）能力**：在精准定界到物理槽位或链路配置的基础上，引入“修复 Agent (Nuwa)”。针对软件配置导致的存储降级，可自动下发修复指令（如重置链路、热插拔扫描）；针对物理坏道，可自动触发业务层的数据迁移与隔离，并联动工单系统派发物理换盘任务，实现“诊断-修复-验证”的全自动闭环。
3. **全栈分布式追踪与自动驾驶**：深化 **轩辕 (Xuanyuan) 总控 Agent** 的调度能力。在分布式存储场景下，实现跨节点的日志关联分析，自动绘制端到端的 I/O 延迟拓扑，进一步迈向存储运维的“自动驾驶”。
