# 智能运维诊断系统架构设计  

## 1. 运维场景面临的挑战

### 1.1 EulerOS 底层运维的复杂性与痛点
在 EulerOS 操作系统的深度运维与大规模集群管理场景下，IT 系统底层环境错综复杂，呈现出以下显著特点：

1. **软硬耦合极深，架构立体复杂**：操作系统承上启下，向下纳管海量异构硬件，向上支撑复杂的微服务与云原生应用，涉及内核态、驱动层与用户态应用的深度交互与依赖。
2. **信息海量且碎片化**：带内日志（In-band Logs，如 Syslog、Dmesg、eBPF/Trace、应用日志等）与带外日志（Out-of-band Logs，如 BMC/IPMI、硬件告警等）散落在不同平台与维度中。在排查时需要跨越带内带外上下文，手工关联海量碎片化线索，耗时耗力。
3. **排查门槛极高，周期漫长**：线上环境如履薄冰。疑难杂症（如内核态死锁、内存缓慢泄漏、TCP 偶发丢包等）高度依赖资深专家基于经验进行“假设-验证”的循环：需要从网络、内存、CPU调度等各个维度反复推敲验证。这种高度依赖个人经验的探索式排查不仅容错率极低（任何不当操作都可能引发二次故障），且排查过程极其复杂，耗时周期极长。
4. **跨域协同壁垒高，MTTR 难以压缩**：操作系统级故障往往牵一发而动全身。在深水区排查时，常常需要拉通内核调度、网络协议栈、存储驱动等多个底层研发团队进行联合诊断，甚至需要深入阅读开源社区源码或进行汇编级分析才能定界问题。这种高复杂度的跨团队协作与源码级深挖导致沟通与验证成本极高，故障恢复时间 (MTTR) 居高不下。

### 1.2 业界主流 Agent 框架在深水区运维的局限性
近年来，以 OpenCode、OpenClaw 为代表的业界主流 Agent 产品与框架展现了强大的通用代码编写与命令执行能力，能够在一定程度上解决基础的、单点的日常运维问题（如查看服务状态、清理磁盘空间等）。但在面对严苛、复杂的 EulerOS 核心运维场景时，仍存在明显的局限性与挑战：

1. **无法适配运维场景的原生特征**：真实的运维环境与纯代码编写环境存在巨大差异，通用 Agent 难以处理以下问题：
   - **输入质量决定排查上限**：在运维领域，故障现象的描述越详细、前置条件越清晰，排查的方向就越收敛，结果也就越准确。然而，通用 Agent 往往习惯于“被动接收并立即执行”，缺乏主动向用户“追问澄清”模糊现象的能力，导致常常在错误的方向上浪费算力。
   - **强时间敏感性**：大部分运维排查高度依赖时间线的对齐（如故障发生前后 5 分钟的指标与日志突变），而通用 Agent 缺乏对时间维度的空间关联能力。
   - **排查方式多样**：运维不仅包含“在线命令交互”，还涉及复杂的“离线分析（如 Dump、离线日志包解析）”，通用 Agent 往往无法有效区分当前处于何种排查场景，导致行动策略混乱。
2. **超长上下文排查导致的严重干扰**：运维排查是一个长周期的证据收集过程。面对动辄数百 MB 的系统日志、网络抓包或复杂的内核输出，**系统中往往存在海量无用或常规日志的高噪音干扰**。这种海量信息的涌入极易掩盖真正的错误线索，导致通用 Agent 的上下文迅速溢出或注意力失焦，在多轮交互中遗忘初始排查目标。
3. **不可忽视的安全隐患**：在极度敏感的操作系统底层（在线场景），让 AI 毫无约束地自由敲击和执行命令存在巨大的操作风险。通用 Agent 往往采用单体黑盒的“一站式”执行，缺乏过程的透明度与安全可控的人工熔断/审核机制。
4. **模型局限性导致的“逻辑死胡同”**：在解决深水区根因问题时，通用大模型由于缺乏严谨的排查 SOP 约束，极易陷入“自我说服”的盲区（即基于片面的证据强行给出一个看似合理的错误结论，并不断围绕该错误结论去寻找支撑证据），难以完成真正的客观推导。

## 2. 我们的破局之道：Multi-Agent 协同诊断框架

面对上述痛点与通用 Agent 的局限性，传统的“静态规则引擎”已力不从心，而“纯对话式的单体大模型助手”又无法在严苛的 EulerOS 运维场景中闭环。为此，我们提出了一种全新的解法：**基于 Multi-Agent 协同的智能运维诊断框架**。

### 2.1 针对通用 Agent 痛点的解决思路
针对 `1.2` 章节中提到的挑战，我们的框架设计了以下针对性的破局思路：
1. **应对运维原生特征与输入的不确定性**：
   - **构建诊断规划与前置澄清**：针对输入模糊和场景多样的问题，引入专门的“规划 Agent (Fuxi)”。在执行任何诊断前，该 Agent 会主动与运维人员进行多轮自然语言交流，澄清并获取必要的前置条件（如：明确是在线诊断还是离线分析、圈定故障发生的精确时间窗口、明确关键现象）。在信息完备后，由其输出结构化的“诊断分析计划 (Plan)”，从源头收敛排查方向并过滤噪音。
2. **克服超长上下文干扰与垂直领域知识匮乏**：
   - **分阶段流水线 (Pipeline) 架构**：摒弃单体模型的长链条推理，通过拆解“诊断规划 -> 并行排查 -> 根因分析 -> 故障修复”四个独立阶段。每个阶段的 Agent 仅提炼和传递关键结论与证据（如排查阶段提取的异常栈），自动屏蔽并丢弃海量无效的中间排查日志，彻底避免上下文传递过程中的注意力失焦与溢出。
   - **专家技能挂载**：系统不再依赖模型裸奔，而是内置了丰富的“故障模式库 (Fault Models)”与 OS 专属“专家技能 (Skills)”。这些 Skills 赋予了模型对带内/带外不同类型复杂日志的结构化理解与深度分析能力，有效弥补了通用模型在垂直领域的知识短板，大幅减少了模型“无头苍蝇式”的乱撞式排查。
3. **消除黑盒执行的安全隐患**：设计了安全可控的 **双模驱动 (Dual-Mode)** 机制与 **权限物理隔离** 策略。
   - **Agent 权限严格隔离**：在“诊断规划、并行排查、根因分析”等前置阶段，Agent 仅拥有只读权限，严禁进行任何系统更改操作；只有流转到“故障修复 (Nuwa)”阶段，才会被赋予变更权限。
   - **人机回环 (Human-in-the-loop) 确认**：在故障修复阶段，Agent 生成的完整修复方案必须主动暂停并交由运维人员审核。只有在运维人员明确授权确认后，系统才会严格、严谨地按照方案执行修复操作，确保线上环境绝对安全。
4. **打破模型自我说服的盲区**：
    - **“假设-验证”闭环**：借鉴业界（如 Google SRE）经典的故障排查方法论，引入“故障现象 -> 故障模式”的映射机制。首先通过现象推导出可能的多个故障模式假设，强制要求结论必须由客观数据支撑，杜绝主观臆断。
    - **多 Agent 并行分析**：在验证阶段，调度多个专职 Agent 并行执行排查任务（如网络 Agent 查丢包、存储 Agent 查 IO 瓶颈）。这种分维度的并发分析不仅大幅缩短了排查时间（MTTR），更保证了复杂故障下各子系统证据的全面收集与交叉验证。
    - **多维综合根因分析**：在获取各并行 Agent 的局部排查证据后，由专门的分析 Agent 介入。它会将这些碎片化的证据串联起来，从“时间线”、“故障调用链路”等全局维度进行综合推演与交叉印证，从而穿透表象，推导出最正确的故障根因。

### 2.2 核心策略落地详解
在上述破局思路的指导下，我们的核心方案策略具体落地为以下四个维度：

- **从“被动响应问答”走向“主动探知执行”**：赋予大模型真正的“手和眼”。系统深度适配 EulerOS 等 Linux 原生环境，Agent 能够像真实的资深运维专家一样，主动连接服务器、执行探测命令、抓取关键日志，甚至深度分析 Dump 文件，实现真正的“知行合一”。
- **分而治之的流水线架构**：我们将复杂的诊断过程科学解耦为 **“规划(Plan) -> 编排(Orchestrate) -> 执行(Execute) -> 分析(Analyze) -> 修复(Repair)”** 五个标准化阶段。每一步都通过结构化的中间件传递上下文，步步为营。
- **安全可控的灵活运行机制**：
  - **分阶段交互模式 (人机回环)**：针对高危环境和疑难场景，系统会在关键决策节点暂停，交由人类专家审核和修正，确保排查方向绝对正确、操作绝对安全。
  - **全自动端到端模式 (自动驾驶)**：针对常见标准化故障，提供“一键式”全自动巡航，从输入现象到输出根因报告全程闭环，极致压缩故障处理时间。
- **专家经验的数字资产化**：将资深专家的排查思路、直觉与方法论，沉淀为系统内置的代码级“技能”。让系统不仅能解决常规问题，更具备攻坚硬核故障的底气，且能够伴随每次实战持续进化。

## 3. 基于 Multi-Agent 协同的安全可控运行机制

> **设计理念**: 运维场景对准确性与安全性的要求极高。为此，系统在底层通过标准化的结构文件 (Plan/Report) 串联起整个诊断流水线，确保证据链的透明与可追溯；在上层则设计了“分阶段交互”与“全自动端到端”两种灵活的运行策略，既能让专家在疑难杂症中安全把控，又能让常见故障的排查一路绿灯，实现严谨性与效率的完美平衡。

### 3.1 分阶段交互模式：安全可控的人机回环 [默认推荐]
针对复杂疑难故障或高危环境，系统提供“人机回环 (Human-in-the-loop)”机制，允许专家在关键节点介入，确保诊断方向正确且操作绝对安全。

```text
├─ 适用场景: 
│   ├─ 故障现象模糊，需要人工确认排查方向
│   ├─ 涉及敏感操作，必须人工审核验证命令
│   └─ 根因推导复杂，需要专家经验辅助决策
├─ 流程特点:
│   ├─ 阶段 1 (诊断规划): 故障环境信息收集与诊断计划生成 -> [系统暂停] -> 专家审核/修改诊断计划
│   ├─ 阶段 2 (并行排查): 基于“假设-验证”调度多 Agent 并发诊断与收集证据 -> [系统暂停] -> 专家确认证据有效性或补充线索
│   ├─ 阶段 3 (根因分析): 基于证据链生成综合分析报告 -> [系统暂停] -> 专家确认根因结论
│   └─ 阶段 4 (故障修复): 生成修复操作方案 -> [系统暂停] -> 专家明确授权后执行修复
└─ 机制支撑: 各阶段状态通过标准文件 (Markdown/JSON) 持久化传递，支持在任意节点随时中断与恢复 (Checkpoint & Resume)。
```

### 3.2 全自动端到端模式：一键式智能巡航
> **设计初衷**: 针对常见故障或标准化的排查流程，提供“一键式”自动化诊断能力，减少人工重复操作，快速定位问题根因。

```text
├─ 适用场景:
│   ├─ 标准化故障排查 (SOP)
│   ├─ 已知模式的快速验证
│   └─ 告警风暴时的批量初筛
├─ 触发方式: 使用 autopilot (全自动巡航) 关键字或直接呼叫 轩辕 (Xuanyuan - 智能总控)
└─ 流程特点:
    ├─ 一键触发: 用户仅需输入指令 (e.g., "autopilot 排查 SSH 连接失败")
    ├─ 全自动流转: 轩辕 接管全流程，自动协调伏羲、大禹、夸父、白泽进行流水线作业
    └─ 结果闭环: 全程无人工干预，直至产出最终诊断报告 (~/.baize/report/*_report.md)
```

## 4. 核心架构与阶段设计

> **设计理念**: 采用流水线式 (Pipeline) 架构，将复杂诊断过程解耦为四个标准化阶段，各阶段 Agent 职责单一、边界清晰。

### 4.3 全链路智能总控实现机制
> **核心定位**: 作为系统的大脑与神经中枢（Xuanyuan Agent），智能总控负责监听用户指令，统筹各阶段 Agent（Fuxi, Dayu, Kuafu, Baize, Nuwa），通过严密的状态机机制协调整个运维流水线的运转与异常接管。

```text
├─ 触发入口:
│   ├─ 关键词监听: 监听用户输入中的 `autopilot`, `auto-diag`, `全自动排查` 等指令
│   └─ 显式调用: 用户直接 @Xuanyuan Agent
├─ 状态机管理:
│   ├─ 空闲状态: 监听模式，等待用户指令输入
│   ├─ 阶段 1 (诊断规划): 唤醒 Fuxi Agent，监控环境信息收集与诊断计划的生成状态，等待确认
│   ├─ 阶段 2 (并行排查): 唤醒 Dayu Agent，将诊断计划拆解为有向无环图，调度多个 Kuafu Agent 并发执行验证，处理超时与异常
│   ├─ 阶段 3 (根因分析): 唤醒 Baize Agent，聚合所有排查证据，进行综合推断并生成最终诊断报告
│   ├─ 阶段 4 (故障修复): 唤醒 Nuwa Agent，基于根因报告生成修复操作方案，严格等待授权后执行修复验证
│   └─ 完成状态: 推送最终处理结果，释放所有 Agent 资源，归档此次故障数据
└─ 异常接管:
    ├─ 若某阶段 Agent 失败，轩辕总控自动降级为“分阶段交互模式”，请求人工介入
    └─ 若执行超时，自动触发熔断，保存当前已完成的中间状态
```

### 4.4 分阶段交互流水线详解

#### 4.4.1 阶段 1：场景识别与诊断建模 (Planning)
> **核心定位**: 识别故障场景与环境上下文，完成信息准入检查，构建初步诊断模型与排查策略。

```text
├─ 1. 场景识别
│   ├─ 在线诊断：获取在线环境的 IP、账号和密码，实时接入系统进行非侵入式探测
│   └─ 离线分析：获取日志路径与类型，基于日志包进行事后分析（若有独立分析环境，也需获取凭证）
├─ 2. 故障澄清与关键信息确认
│   ├─ 核心职责：通过交互式提问还原故障现场，澄清模糊的故障描述
│   ├─ 关键要素确认：明确故障发生的准确时间、具体的故障现象
│   └─ 准入检查：验证信息是否完整，避免在缺乏上下文时盲目诊断
│       ├─ 明确故障对象：确认具体的组件、进程或系统模块
│       ├─ 明确时间窗口：确认是当前正在发生的故障，还是历史追溯及具体时间段
│       └─ 确认现象可观测性：
│           ├─ 在线特征：确认是否有具体的报错日志、监控指标异常或内核堆栈
│           └─ 离线数据：明确具体的日志类型（Syslog、Dmesg、业务日志）或 Dump 类型（Vmcore、Java Heap Dump）
├─ 3. 诊断可行性评估
│   ├─ 在线诊断：
│   │   ├─ 免密配置：尝试基于凭证配置免密登录
│   │   └─ 环境探测：仅执行登录验证和查询系统基础版本信息。严禁在此时执行任何故障检测命令
│   └─ 离线分析：
│       ├─ 远程分析服务器：配置免密登录并校验日志路径是否存在
│       └─ 本地日志：直接校验本地日志路径的有效性
└─ 4. 诊断模型构建
    ├─ 核心目标：整合前期信息，构建“现象 -> 潜在故障模式”的假设树，生成标准化诊断排查计划
    ├─ 模型构建逻辑：
    │   ├─ 方法论指导：遵循“现象驱动 → 假设收敛 → 验证闭环”的核心思想
    │   ├─ 构建原则：故障模式需相互独立且穷尽，此时不推断最终根因，仅收敛出潜在的故障模式与排查方向
    │   ├─ 假设树生成：以故障现象为根节点，推导出多个可能的故障模式假设（如 CPU 饱和、内存泄漏等）
    │   └─ 收敛策略：计算并输出 Top 5 可能性最高的故障模型假设
    └─ 计划内容输出：
        ├─ 存储路径：`~/.agent/plans/{timestamp}_{plan_id}.md`
        └─ 计划内容结构：
            ├─ 1) 故障场景：记录在线/离线模式及对应的连接凭证信息
            ├─ 2) 故障澄清：记录用户原始描述与经过确认的完整故障现象
            ├─ 3) 前期检测结果：记录环境可达性、基础环境信息与数据完备性
            └─ 4) 诊断模型：基于假设树生成按优先级排序的验证任务表格
                
                - [关键机制] 必须在文件末尾附加 JSON 格式的任务元数据，供阶段2调度解析：
                  ```json
                  {
                    "plan_id": "20240320_001",
                    "tasks": [
                      { "id": "T1", "mode": "CPU 饱和", "verification_target": "进程资源消耗", "priority": "P1" }
                    ]
                  }
                  ```
```

#### 4.4.2 阶段 2：基于假设的并行排查与证据收集
> **核心定位**: 基于阶段 1 生成的“假设树”，将验证任务拆解并分发给多个专职 Agent 进行并发探测与证据提取，大幅缩短排查时间。

```text
├─ 1. 任务分发策略
│   ├─ 双模输入机制：
│   │   ├─ 直接输入模式：接收用户直接下发的自然语言指令（如“排查CPU高”），由调度 Agent 实时拆解为临时任务，不依赖计划文件
│   │   └─ 计划执行模式：读取阶段 1 输出的结构化排查计划文件，解析文件末尾的 JSON 任务元数据，批量加载排查任务
│   └─ 并发路由策略：
│       ├─ 任务统一封装：将所有来源的输入统一转换为标准化的 Task 对象
│       ├─ 按需分发执行：遍历 Task 列表，根据任务类型将任务路由给最匹配的执行 Agent
│       └─ 独立会话隔离：采用完全并行的调度策略，每个排查任务均在一个隔离的 Agent 会话中独立运行，互不干扰
├─ 2. 诊断执行引擎 (通用与专项探测)
│   ├─ 通用分析 Agent：
│   │   ├─ 核心定位：作为一线全科排查者，负责标准化指标的验证与浅层证据分析
│   │   ├─ 执行逻辑：调用标准系统工具（Skills）进行宽泛探测；一旦发现异常深层数据，立即触发专家会诊机制
│   │   └─ 联动示例：当探测到存在 /var/crash/vmcore 文件时，自动唤醒并委托 Vmcore 分析插件接管
│   └─ 专用分析 Agent (专家会诊)：
│       ├─ 核心定位：作为二线专科排查者，专注攻坚硬核或顽固类故障（如内核崩溃、内存缓慢泄漏、分布式死锁等）
│       └─ 诊断示例：Vmcore Agent 负责解析内核 Dump 以定位 Panic 根因；Java 堆栈 Agent 负责深挖 OOM 对象引用链
└─ 3. 证据收集与标准化归档
    ├─ 全局状态追踪：实时监控并管控所有并发 Agent 的执行进度与健康度
    ├─ 证据标准化输出：将各 Agent 提取的非结构化线索，统一清洗并转换为结构化的证据对象
    └─ 结果持久化汇聚：待所有任务节点执行完毕后，将汇总的验证结果集持久化至标准的 JSON/Markdown 报告中，供下一阶段分析使用
```

#### 4.4.3 阶段 3：多维结果汇总与根因推断
> **核心定位**: 通晓系统原理与异常模式，穿透表象识别真实根因，关联告警、日志、指标形成完整证据链。

```text
├─ 1. 证据聚合与上下文关联
│   ├─ 多源数据汇总：拉取阶段 2 中所有并行 Agent 输出的结构化诊断结果
│   ├─ 时间线重建：基于统一的时间戳，将分散在不同日志和组件中的异常事件对齐，还原故障发生的先后顺序
│   └─ 证据链构建：梳理各异常事件之间的因果关系，形成从“表面现象”推导至“底层根因”的完整逻辑链路
├─ 2. 根因推断与验证
│   ├─ 交叉验证：利用多维度的证据（如 CPU 飙高与特定进程的系统调用栈）进行相互印证，避免单一视角的误判
│   ├─ 噪音排除：过滤掉由主故障引发的次生告警或无关的常规系统事件
│   └─ 置信度评估：为推导出的每个潜在根因打分，锁定最符合当前证据链的唯一真实根因
├─ 3. 故障影响面评估
│   └─ 范围界定：基于根因定界，评估该故障对其他业务模块、网络节点或系统资源造成的连带影响程度
└─ 4. 诊断报告生成
    ├─ 报告组装：整合阶段 1 的场景信息、阶段 2 的排查证据以及本阶段的根因分析结论
    ├─ 格式化输出：生成结构清晰、人类可读性强的最终诊断报告（包含故障现象、时间线、证据链、根因结论）
    └─ 持久化存储：将报告保存至 `~/.agent/report/{timestamp}_{plan_id}_report.md`，准备移交修复阶段
```

#### 4.4.4 阶段 4：解决方案生成与智能修复
> **核心定位**: 基于根因报告提供止血与根治方案。本阶段是整个流水线中**唯一被赋予系统更改权限**的环节，所有操作严格遵循“方案必确认、执行必记录、风险可回滚”的安全底线。

```text
├─ 1. 安全修复方案生成
│   ├─ 方案推荐：提供即时止血方案（快速恢复业务）与根治方案（彻底修复底层 Bug 或配置）
│   ├─ 前置回滚机制：所有修复方案必须强制包含对应的“环境快照备份步骤”与“一键回滚脚本”
│   └─ 风险评估：对每一步修复操作可能引发的系统波动进行评级，并清晰展示给运维人员
├─ 2. 人机回环与授权控制
│   ├─ 方案预审：Agent 生成方案后，系统必须暂停，将详细的操作步骤推送给用户进行二次确认
│   └─ 权限释放：只有在获取用户的明确授权指令后，修复 Agent 才会被临时授予编辑文件或执行变更命令的权限
├─ 3. 严谨执行与审计闭环
│   ├─ 步骤化执行：严格按照预审通过的步骤逐条执行，每执行一步均需验证返回结果是否符合预期
│   ├─ 操作全量审计：在执行过程中，Agent 下发的所有 Shell 命令及其输出结果将被完整记录，形成不可篡改的审计日志
│   └─ 修复效果验证：执行完毕后，自动调用相关的探针验证故障是否消除；若验证失败，则提示并辅助用户触发回滚
└─ 3. 知识沉淀
    └─ 将本次成功的“故障现象 -> 根因 -> 修复方案”完整链路存入专属知识库，用于后续同类故障的快速匹配
```

## 5. 架构总结与未来规划

### 5.1 架构核心优势总结
本框架通过引入 **Multi-Agent 协同** 与 **流水线 (Pipeline) 架构**，彻底重塑了传统依赖单体大模型“对话式”排查的模式，在复杂 OS 运维场景下实现了以下突破：
1. **破解了单体 Agent 的场景感知缺失**：通过伏羲 (Fuxi) 的前置澄清，实现了模糊现象到具体诊断目标的精准转译，避免了排查方向的南辕北辙。
2. **克服了超长上下文与高噪干扰**：大禹 (Dayu) 与夸父 (Kuafu) 的“基于模式的定点提取与隔离机制”，彻底解决了海量系统日志、GB 级 vmcore 文件导致的上下文溢出与大模型幻觉问题。
3. **打破了逻辑盲区与浅层归因**：白泽 (Baize) 引入的软硬结合时序推演与强证据链（Evidence Chain）机制，实现了从表象报错到确切物理槽位或底层源码级缺陷的穿透，极大压缩了跨团队协同的 MTTR。
4. **实现了专家经验的数字资产化**：通过标准化的 Skills 挂载，使得 Agent 能够像资深内核专家一样，熟练运用 `crash`、`ibmc` 等底层工具，不再“纸上谈兵”。

### 5.2 未来演进方向：面向运维原生的极致安全与自愈
随着系统的不断落地与实战数据的积累，在跑通了“现象->定界->根因”的诊断闭环后，系统的下一步演进将从“会诊断”迈向“敢执行”，全面拥抱极度严苛的生产环境运维标准：

1. **更严密的安全执行边界（Safe Execution）**
   - **指令级沙箱拦截**：从架构底层建立“高危操作拦截网”，对于任何试图修改系统内核参数、重启核心服务或格式化磁盘的指令，实施基于规则引擎的物理级拦截与预警。
   - **基于 RBAC 的权限收敛**：细化 Agent 在不同环境下的权限粒度。日常诊断仅授予只读权限，只有在特定的修复节点且经过多重身份校验后，才临时授予有限的变更权限。

2. **全链路的操作可追溯（Traceability & Audit）**
   - **防篡改审计日志**：Agent 所有的环境探测、工具调用、下发到远端服务器的脚本执行，都将被赋予唯一的 Trace ID，并记录到防篡改的审计平台中。
   - **“为什么这么做”的透明化白盒**：不仅仅记录“做了什么”，还要持久化保存 Agent 当时的推理逻辑（Prompt 上下文与推导树），确保在事后复盘时，人类专家能 100% 还原 AI 的决策过程。

3. **原生级的可回滚与自愈保障（Rollback & Self-healing）**
   - **变更前的“快照”强制约束**：在女娲 (Nuwa) 执行任何修复动作前，强制要求生成系统状态快照或配置备份。
   - **基于校验的自动回滚闭环**：修复执行后，自动触发后置校验（Post-check）。一旦发现系统核心指标发生恶化，系统能够依据先前的快照，毫秒级触发自动回滚机制。

4. **全链路拓扑感知与 RAG 知识库融合**
   - 引入基于图数据库的系统依赖拓扑，结合 RAG（检索增强生成）技术，让 Agent 在规划和分析时能够实时关联历史相似故障 SOP 与业务拓扑影响面，实现从“单点排查”到“全局视域”的跨越。

**从辅助排查的“智能副驾”，到主导自愈的“自动驾驶”，witty-diagnosis-agent 正以敬畏生产环境的态度，稳步重塑下一代底层运维范式。**
```
