## 一、 总览：什么是 Evals 评测体系？（定位与价值）

过去，我们评测一个模型，往往是喂给它一段文本，看它生成的回复顺不顺畅（单轮 Prompt 评测）。但这对于能够自主调用工具、执行多步推理的 Agent 来说，是远远不够的。

OpenAI Evals 体系的**核心定位**发生了一次重大转变：
- **过去**：衡量模型能力的“期末考试”（偏研究性质）。
- **现在**：贯穿 Agent 全生命周期的“工业质检流水线”（偏工程应用）。它不仅看最终输出结果，还要深入评估 Agent 的**执行轨迹（比如调用了什么 API、传了什么参数）**。

**Evals 在开发中的位置：**
就像软件工程中写代码前先写单元测试一样，Evals 现在是 Agent 开发的**标准起点**：
`定义评测标准（写考卷） -> 编写 Prompt 与工具（学知识） -> 运行评测（考试） -> 发现错误自动优化（补习） -> 部署上线（实战）`

## 二、 核心组件：评测体系的“五脏六腑”（What）

要搭建一套完整的评测体系，你需要了解以下四个核心“角色”：

### 1. 考卷：数据集（Datasets）
这是评测的原材料。考卷的题型非常丰富：
- **标准问答对**：一问一答（适合信息提取、分类）。
- **工具调用序列**：输入问题，期待 Agent 按照特定顺序调用指定工具。
- **多轮对话**：给一段历史聊天记录，期待 Agent 做出得体的回应。
*（建议：冷启动准备 50-100 道题找典型错误，成熟期扩充到 500+ 道题做精准打分）*

### 2. 裁判：自动化评分器（Graders）
谁来批改这 500 道题？OpenAI 提供了三类自动化阅卷老师（详见“深度挖掘”章节）：
- **String Check（字符串匹配）**：最死板的老师，只认死理（比如必须输出“是”或“否”）。
- **Python Grader（自定义代码）**：懂逻辑的老师，通过写 Python 脚本来验证 JSON 格式或计算数值。
- **LLM Judge（大模型裁判）**：最聪明的老师，让另一个更强的大模型（如 o3）来评估语气的专业度、逻辑的连贯性。

### 3. 显微镜：轨迹评分（Trace Grading）
这是 Agent 评测专属的“黑科技”。它不仅给最终答案打分，还会把 Agent 的思考过程、工具调用过程拆解开来，**对每一步分别打分**。

### 4. 辅导老师：提示词优化器（Prompt Optimizer）
这是一个自动化工具。当系统发现你的 Agent 经常在某类问题上失分时，优化器会自动分析失败原因，并**帮你重写或微调 Prompt**，然后预测新 Prompt 能提分多少。这就把“玄学调参”变成了“数据驱动”。

## 三、 设计哲学：为什么必须做复杂评测？（Why）

### 1. 拒绝盲目调参：没有测量就没有改进
如果你没有一套 Evals 系统，你永远不知道：
- 把模型从 GPT-4 换到 GPT-4.5，业务指标是真的提升了，还是在某些场景下退步了？
- 你熬夜加的两句 Prompt，到底是解决了 1 个 bug，还是引入了 10 个新 bug？

### 2. 测试驱动开发（BDD）的 AI 翻版
传统软件开发提倡 BDD（行为驱动开发）。OpenAI 将这一理念平移到了 Agent 开发上：**不要先急着写 Prompt，先写 Eval（评测标准）。** 只有当你能清晰地定义出“什么是 100 分的回答”，你才能做出好产品。

### 3. 过程比结果更重要
传统评测只看最终答案，这在 Agent 时代会掩盖大量隐患。
*比如：如果 Agent 最终回答对了，但中间为了找答案，它把同一个接口循环调用了 50 次，浪费了大量 Token 成本。传统评测会给满分，但轨迹评测（Trace Grading）会亮起红灯。*

## 四、 运转流程：让 Agent 进化的“评测飞轮”（How）

OpenAI 提出了一套**“评测飞轮”（Evaluation Flywheel）**，这是一个让 Agent 越用越聪明的闭环流程：

1. **找错题（错误分析）**：从线上真实用户的提问中，抽样 50 条回答得不好的案例，人工分类（比如：35%是没调用日历工具，20%是语气太生硬）。
2. **出考卷与定标准（构建 Dataset 与 Grader）**：把这些错题做成测试集，并配置好“裁判员”（比如写一段 Python 代码来检查日历工具的参数）。
3. **摸底考（建立基线）**：跑一次评测，记录当前得分（比如：工具调用正确率 72%）。这个分数就是“基线”。
4. **补习提升（优化与验证）**：人工或使用 Prompt Optimizer 修改提示词，再考一次。如果分数提升到 88% 且没有引发其他问题，则通过。
5. **常态化考试（CI/CD 集成）**：把这套考试系统接入代码流水线，以后任何同事修改了代码或模型，都必须自动跑一遍这 50 题，及格了才能上线。

## 五、 典型应用场景：什么时候该用什么评测？（Where）

1. **离线评测（Offline Evals）**
   - **什么时候用**：模型大版本升级、Prompt 大改版、验证基础的分类/信息提取能力。
   - **怎么用**：在本地或控制台跑几百个测试用例，对比新旧版本的得分。
2. **轨迹评分（Trace Grading）**
   - **什么时候用**：调试多步协作的复杂 Agent（如自动订票助手、写代码 Agent）。
   - **怎么用**：不仅看它最后有没有订票成功，还要看它有没有先查库存、再核对用户身份、最后再下单。
3. **人机混合评审（Human-in-the-loop）**
   - **什么时候用**：医疗诊断建议、法律合同起草、高级营销文案撰写。
   - **怎么用**：机器先打分过滤，最后一道关卡由人类领域专家盲测（不知道是哪个模型写的），确保高风险场景的安全与主观题的质量。

## 六、 深度挖掘（Deep Dive）：难点剖析与通俗示例

> **分析：我能否看懂这些高阶概念？**
> 答：完全可以。其实这套体系本质上就是把人类学校里的“教务处考核系统”搬到了 AI 身上。为了让大家更通俗易懂，我梳理了以下几个核心难点，并给出了大白话示例。

### 1. 评分器（Grader）怎么选最划算？
阅卷老师是有成本的，不能一刀切：
- **String Check（字符串匹配）**：**极快、免费**。适合做选择题（比如情感分析，输出“正向/负向”）。但它太笨，如果标准答案是“开心”，模型回答“高兴”，它会判错。
- **Python Grader（自定义逻辑）**：**较快、低成本**。适合做填空题和格式题（比如验证模型输出的 JSON 里有没有包含 `order_id` 这个字段，日期格式是不是 `YYYY-MM-DD`）。
- **LLM Judge（大模型裁判）**：**慢、费钱**。适合做作文题（比如评估客服回答是否温柔体贴）。

### 2. 为什么“轨迹评分（Trace Grading）”是 Agent 评测的灵魂？
**【通俗案例：查快递 Agent】**
假设用户问：“我的订单 12345 到哪了？”

* **情景 A（蒙对的学渣）**：Agent 脑子一抽，没有调用 `get_order_status` 工具，而是直接编造了一个回答：“您的订单正在运输中，明天送达。”（碰巧明天真的送达了）。
  - **传统评测结果**：100分（答案对了呀！）。
  - **Trace 评测结果**：0分（严重警告：出现幻觉，未按规定路径调用工具）。

* **情景 B（磨叽的学渣）**：Agent 调用了 `get_order_status(id=123)`，报错了；又调用 `get_order_status(id=1234)`，又报错了；试了 5 次才终于输对 `12345`。
  - **传统评测结果**：100分（最终查出来了，答案正确）。
  - **Trace 评测结果**：60分（扣掉调用效率分，存在冗余消耗）。

**结论**：Trace Grading 就是为了抓住那些**“结果对了但过程很扯”**的隐患，这是 Agent 稳定落地的命门。

### 3. 用大模型当裁判（LLM Judge）的“偏见”挑战
让 AI 批改 AI 的卷子，会有明显的“偏见”：
- **位置偏见（Position Bias）**：如果是 A/B 两个回答让它选，它往往觉得先看到的 A 更好。
  - *解法*：代码里写个随机数，一半概率把 A 放前面，一半概率把 B 放前面。
- **长篇大论偏见（Verbosity Bias）**：它觉得字数越多的回答显得越专业。
  - *解法*：在 Prompt 里严格规定“啰嗦扣分”，或者先把超长文本截断再让它评分。
- **自我偏爱**：GPT 模型当裁判时，潜意识里更喜欢 GPT 生成的风格。
  - *解法*：尽量使用推理能力极强且更客观的模型（如 OpenAI o3）来做 Judge，并强制要求它“先写出扣分理由，最后再打分（Chain-of-Thought）”。

## 七、 总结与行动指南：给开发者的落地建议

构建 Agent 评测体系不是一蹴而就的，建议分为三步走：

1. **起步期（0-1个月）：先别搞太复杂，把“错题本”建起来。**
   - 不要一上来就用 LLM Judge。先找 50 个线上的真实错误案例，写几个最简单的 Python Grader。
   - 确立你的“基线分数”，哪怕只有 60 分，也比瞎蒙强。
2. **成长期（1-3个月）：让“考试”常态化。**
   - 把评测接入 CI/CD，定下规矩：任何人修改了 Agent 的 Prompt 或工具代码，必须跑通这 50 个测试用例。
   - 开始引入 Trace Grading，监控工具调用的准确率。
3. **成熟期（3个月以后）：让“飞轮”自己转起来。**
   - 使用 Prompt Optimizer 自动优化提示词。
   - 每天自动从线上抽样 5% 的聊天记录让 LLM Judge 打分，低于 3 分的自动报警并加入测试集。

**终极洞察：**
OpenAI Evals 体系最大的价值，并不是提供了一堆时髦的代码工具，而是**倡导了一种严谨的工程文化**。当你的团队能够自信地说出：“我知道我们的 Agent 在哪里会犯错，并且我们有一套流水线能让它明天不再犯同样的错”时，你就真正拥有了 AI 时代的竞争壁垒。
