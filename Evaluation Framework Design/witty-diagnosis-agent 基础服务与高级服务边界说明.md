## 一句话定位

基础服务回答"哪里出了问题、为什么"；高级服务在此基础上，通过专有工具集成、鲲鹏专项场景包、安全审查模型、诊断经验持续沉淀以及主流 Agent 运行时生态兼容，将诊断的**深度、广度、安全性、长期进化能力与生态融合度**提升到生产级要求。

---

## 基础服务（开源）

**核心架构**：五个 Agent 分工协作——Fuxi 负责生成排查计划，Dayu 并发调度，Kuafu 深入节点执行采集与推理，Baize 汇总证据链输出根因报告，Nuwa 生成并执行修复动作。同时，内置 16+ 基础及典型 OS 运维场景 Skill，覆盖常用的诊断和信息收集能力。诊断阶段严格只读，修复阶段需用户审批确认。

依托上述架构，基础服务当前已覆盖以下典型场景（深度结合 openEuler）：

- **离线日志分析**：支持导入包含系统日志、应用日志或 VMCore 等文件的离线数据包，由 Agent 自动解析、错误回溯与关联诊断。涵盖**磁盘故障诊断**、**VMCore 分析**等典型场景，适合脱机或隔离环境下的故障复盘。
- **在线分析诊断**：在生产环境中直接运行，自动采集实时节点数据，进行多维度关联分析，包括磁盘 I/O、网络丢包、内存碎片等资源瓶颈，以及进程死锁、僵尸进程等逻辑异常的即时定位。

## 高级服务（商业授权）

在基础服务完整诊断能力之上，高级服务沿五个核心方向突破现有边界：

| 增强方向 | 核心目标 | 关键能力与解决的痛点 |
| --- | --- | --- |
| **一、场景扩展** | 提升端到端的全智能故障诊断环境配置、数据采集、根因分析流程 | 聚焦鲲鹏训练/推理等专项场景，解决现场环境复现难、性能瓶颈跨层定位难、环境依赖复杂等五大核心难题 |
| **二、工具能力升级** | 基于专有商业化工具，提供专有 Agent 智能分析能力，让工具更智能 | 深度集成 iBMC、eSight、FusionDirector、X-diagnosis 等工具，将纯被动采集数据转化为结构化诊断证据链 |
| **三、自动化执行安全审查能力升级** | 对模型生成的执行命令或自动化脚本进行二次智能过滤与拦截，确认其安全性与合理性 | 引入专属微调大模型（witty-llm），精准拦截硬编码密钥、越权操作、恶意注入等自动化高危场景，杜绝次生灾害 |
| **四、自进化诊断** | 将每次诊断结果自动沉淀为可检索的结构化知识，并在历史诊断间建立关联，实现越用越准的持续进化与主动故障预测 | 诊断结果自动入库，RAG 驱动的知识检索增强后续诊断；跨会话的诊断序列关联分析，支持相同服务器或同类问题的趋势洞察与故障预警 |
| **五、生态兼容与 Agent 运行时扩展** | 打破单一 Agent 运行时锁定，快速接入主流 Harness Agent 框架，最大化诊断能力的生态覆盖面 | 支持 openclaw、hermes、jiuwenclaw 等主流 Agent 框架的标准化适配层，让诊断能力以插件形式无缝嵌入客户现有 Agent 基础设施 |

### 方向一：场景扩展

基础服务的诊断场景覆盖 openEuler 通用 OS 层，高级服务在此之上新增专项场景包，以闭源二进制工具的形式按需加载。其核心目标是**提升端到端的全智能故障诊断环境配置、数据采集、根因分析流程**，当前重点聚焦于**鲲鹏模型训练/推理场景专项诊断**，旨在解决该领域中普遍存在的五大核心难题：

| 难题 | 核心痛点 | 高级服务解法 |
| --- | --- | --- |
| **一、进程 Crash — 客户现场无法定位** | 环境复现难、调试工具受限，定位过程极其耗时 | 在客户侧建立 coredump 自动采集机制，将核心数据打包回传至专项环境，配合鲲鹏专属诊断脚本还原调用栈，实现免现场复现的高效定位 |
| **二、性能瓶颈 — 根因归因困难** | 劣化根因分布在算子效率、内存带宽（鲲鹏 L3 / HBM）、NUMA 跨节点访问、HCCL 通信拓扑四个层级，跨层关联难 | 构建"算子 profiling → 系统 PMU → 网络拓扑"分层联合诊断能力，提供统一的跨层关联视图 |
| **三、知识独有且分散 — 依赖专家口耳相传** | 专家经验难以结构化传承，新人上手门槛极高 | 建立结构化专属知识库（故障特征、版本指纹、根因结论、修复路径），引入 RAG 能力，输入报错日志即可自动匹配历史案例 |
| **四、环境依赖复杂 — 版本组合爆炸** | OS 内核 + 驱动 + CANN + MindSpore/PyTorch-NPU + 模型代码版本组合极多，一次升级即可引发跨层兼容性断裂 | 提供"软件栈指纹"自动采集工具和版本兼容矩阵数据库，配合最小复现环境容器快照，支持专项实验室精确重现 |
| **五、可观测性不足 — 关键故障信号被掩盖** | 数百进程日志分散多节点且时序未对齐，NPU ECC 错误、驱动超时、通信超时等关键信号淹没在海量正常输出中 | 构建统一日志聚合和故障信号提取规则引擎，将底层硬件告警（温控降频、内存 UE）与上层训练精度、吞吐量指标深度联动，实现端到端精准溯源 |

每个专项场景包含专属工具、规则集和 E2E 诊断剧本，不依赖基础服务已有 Skill，独立扩展。

### 方向二：工具能力升级

基础服务的 Kuafu Agent 执行阶段依赖内置采集器与 Skill。高级服务为 Kuafu 扩展了一套**内部自研的闭源二进制工具包**，以及对华为生态典型硬件和管理工具的深度集成。其核心目标是**基于专有商业化工具，提供专有 Agent 智能分析能力，让工具更智能**。

具体来说，高级服务能够对接和消费以下专属工具采集的数据，将原本缺乏深度分析能力的"纯数据"转化为可用于根因诊断的证据链：

| 工具组件 | 数据采集能力 | 高级服务增量价值 |
| --- | --- | --- |
| **X-diagnosis** | 依托 eBPF 等内核特性，非侵入式捕获纳秒级异常（CPU 调度、I/O、网络协议栈、内存）及离线 VMCore 解析 | 作为高级服务的"手和眼"，将碎片化底层异常转化为结构化证据链，实现从内核抖动到业务影响的智能定界 |
| **iBMC 固件** | 采集带外硬件实时状态（温度、电压、电源、SEL 事件日志） | 补足 OS 无法感知的底层线索，实现软硬跨层根因分析 |
| **eSight Agent** | 通过 SNMP/IPMI 等协议上报设备运行指标 | 对接海量运行指标，实现跨设备的异常关联诊断 |
| **FusionDirector** | 汇聚服务器集群资产视图、固件版本和硬件预警 | 将单点故障放入集群生命周期和批量管理中进行综合研判 |
| **A-Tune** | 持续采集 CPU、内存、IO、网络等负载特征画像 | 提供精准业务基线对比，助力性能劣化与异常瓶颈定位 |

这些工具采集能力强但自身分析能力极弱（仅限阈值告警或参数匹配）。高级服务将这些工具输出的原始指标标准化，作为 Baize Agent 根因融合的高质量输入，从而在 openEuler 环境中完成从"被动采集"到"深度智能诊断"的完整闭环。专属工具包以二进制分发，具备签名校验，不可脱离 Agent 单独使用。

### 方向三：自动化执行安全审查能力升级（基于模型微调）

在自动化运维中，故障修复阶段最核心的痛点是"高危命令与恶意脚本"引发的次生灾害（如删库、越权、配置覆写）。基础服务依赖通用 LLM 生成修复动作，缺乏深度的安全审查。高级服务引入了内部专项微调的安全审查模型（witty-llm），其核心目标是**对大模型生成的执行命令或自动化脚本进行二次智能过滤与拦截，确认其安全性和合理性**。

该模型针对**脚本与自动化安全（Script & Automation Security）**进行了深度强化，具备强大的 **Shell 命令及脚本理解与恶意命令分析**能力，能够精准识别和拦截以下高危场景：

- **高危运维命令的智能确认**：对大模型生成的单条或组合修复命令进行二次研判，识别其潜在破坏性并请求必要的人工二次确认；
- **高危命令执行前拦截**：在实际下发至节点前，精准拦截如高危删库、大范围越权、网络配置覆写等高风险操作，实现前置阻断；
- **Skill 内置执行脚本的深度扫描**：对复杂场景包或扩展 Skill 中包含的自动化脚本进行安全审查，识别硬编码密码、恶意指令注入等隐蔽威胁；
- **运维标准流程检测与合规审计**：结合企业内部安全规范，判断自动生成的执行路径和参数是否符合标准运维流程，避免因不当操作引入合规风险。

此外，该模型对"无充分证据时明确拒绝结论"有严格的训练约束，进一步降低生产环境的误判与误操作风险。私有化部署客户可在本地运行量化版模型，确保核心运维数据完全不出境。

### 方向四：自进化诊断

基础服务每次诊断均为独立闭环，诊断结论不留存、不积累，无法从历史中学习，也无法感知同一环境中反复出现的潜在规律。高级服务引入**自进化诊断**机制，将每一次诊断过程转化为持续进化的资产，实现"越用越准"的长期能力增长。

**能力一：诊断结果自动入库，RAG 驱动持续增强**

每次诊断完成后，Baize Agent 自动将根因结论、证据链、修复动作及验证结果结构化后写入专属知识库。知识条目包含关键维度：故障现象特征、触发条件、受影响的软硬件版本指纹、根因分类、已验证的修复路径。后续诊断中，Fuxi Agent 在生成排查计划时会优先检索该知识库，将匹配到的历史案例作为高置信度参考依据注入推理上下文，从而缩短定位路径、提升首次定位准确率。该机制与方向一中的鲲鹏专属知识库相互补充：专属知识库提供专家预置的静态基线，诊断沉淀库则持续积累来自真实生产环境的动态经验。

**能力二：跨会话诊断关联，趋势洞察与故障预警**

高级服务在诊断记录之间建立多维关联索引，支持以下三类关联分析场景：

- **同一服务器的纵向关联**：自动串联同一节点的历史诊断序列，识别周期性劣化规律（如每隔 N 天出现 I/O 抖动、内存碎片持续累积），在问题演变为严重故障之前提前预警。
- **同类问题的横向关联**：跨节点识别相同根因的诊断记录，辅助判断是否存在批量性的版本缺陷、硬件批次问题或配置漂移，提升集群级故障的收敛效率。
- **定时任务诊断的时序关联**：对定期执行的巡检任务（如每日磁盘健康检查、每周性能基线采集），自动建立时序对比视图，量化关键指标的变化趋势，在阈值告警触发之前即给出劣化预测。

三类关联分析的输出统一呈现为**故障趋势报告**，涵盖：当前健康评分、历史劣化曲线、预测故障窗口及建议的预防性干预措施，为运维团队提供从"被动救火"转向"主动防御"的决策依据。

### 方向五：生态兼容与 Agent 运行时扩展

基础服务的五 Agent 协作架构深度绑定自有运行时调度层，在已有独立 Agent 基础设施的企业客户中存在集成摩擦——客户需要在两套 Agent 运行时之间维护割裂的工具链与任务编排，运维复杂度显著上升。高级服务为此引入**标准化 Agent 运行时适配层（Agent Runtime Adapter，ARA）**，其核心目标是**快速接入主流 Harness Agent 框架，让诊断能力以插件形式无缝嵌入客户现有 Agent 基础设施，最大化生态覆盖面**。

**当前重点适配的主流框架：**

| 框架 | 类型定位 | 适配方式 | 集成价值 |
| --- | --- | --- | --- |
| **openclaw** | 开放式多 Agent 编排框架 | 以标准 Tool Provider 形式注册，Fuxi/Baize 能力暴露为 openclaw 原生可调用工具节点 | 客户无需迁移现有 openclaw 工作流，诊断 Skill 直接嵌入编排图谱 |
| **hermes** | 事件驱动型轻量 Agent 框架 | 通过 hermes Event Hook 接入，故障信号触发后自动拉起诊断子 Agent，结果以结构化事件回写 hermes 总线 | 实现告警 → 诊断 → 根因入库的全链路事件驱动自动化，零人工干预 |
| **jiuwenclaw** | 企业级知识增强型 Agent 框架 | 将高级服务知识库（方向四）注册为 jiuwenclaw 外部知识源，诊断上下文与 jiuwenclaw 推理链深度融合 | 借助 jiuwenclaw 原生 RAG 调度能力进一步增强历史案例检索精度 |

除上述三类框架外，ARA 层基于**开放适配协议（Open Adapter Protocol，OAP）**设计，具备以下通用扩展能力：

- **统一 Tool Schema 标准**：诊断工具调用参数与输出格式遵循 JSON Schema + OpenAPI 规范，任意支持 Function Calling 或 Tool Use 协议的 Harness Agent 框架可在数小时内完成接入。
- **运行时隔离沙箱**：各框架适配层在独立进程沙箱中运行，互不干扰，单一框架的异常不影响其他运行时与核心诊断引擎的稳定性。
- **授权粒度穿透**：ARA 层继承高级服务的 License 授权管理机制，按接入框架、节点数量、诊断场景包进行细粒度授权控制，防止能力越权调用。
- **双向上下文同步**：外部框架发起的诊断任务可将自身工作流上下文（如当前告警、变更记录、业务拓扑）注入诊断推理链，诊断结论也可反向推送至外部框架的状态存储，实现跨系统的诊断闭环。

该方向与方向四的自进化诊断形成协同：来自外部 Agent 框架触发的诊断任务，其结果同样会自动入库并参与跨会话关联分析，使"外部集成触发的诊断"与"内部主动巡检的诊断"共享同一套知识积累飞轮。

## 高级服务技术实现分析

高级服务的技术实现本质上解决三个核心问题：**代码怎么隔离、能力怎么交付、模型怎么部署**。这三者相互咬合，任何一个环节设计不当，都会影响商业化壁垒的建立。以下表格总结了各环节的实现策略及其建立的竞争壁垒：

| 实现维度 | 核心策略 | 具体实现机制 | 商业壁垒与防范风险 |
| --- | --- | --- | --- |
| **一、代码隔离**<br>*(双仓库策略)* | **接口开源，实现闭源** | 开源 `witty-diagnosis-agent` 维持现状；高级服务单独建私有库 `witty-diagnosis-enhanced`。在开源端定义标准的**工具调度接口（Tool Runner）**和**场景包加载接口（Scene Loader）**，两库仅通过接口通信。 | **防反弹风险**：必须在高级工具开发前提前设计好开源接口，避免日后强行拆分引发社区反弹。<br>**生态兼容**：接口公开允许社区开发插件，同时无缝接入闭源高级工具包。 |
| **二、能力交付**<br>*(专有 Agent 与工具包)* | **二进制专有 Agent，按需加载，防篡改** | 核心采集与底层诊断能力以**专有 Agent**（闭源二进制）的形式交付，加载专有的 Skills（含专属规则集、E2E 剧本、Prompt 模板），提供有限的扩展能力。<br>规范统一的调用参数格式与 JSON Schema 输出。 | **防伪造/篡改**：通过 Ed25519 验证二进制签名，并内嵌 RSA 签名的 License 文件进行细粒度授权（按节点/场景/有效期）。<br>**极高壁垒**：鲲鹏专有 Skills 需直接读取 aarch64 硬件计数器并调用专属系统事件，此类逻辑在 x86 上无法运行，社区极难复制。 |
| **三、模型部署**<br>*(本地 SLM)* | **不公开数据集的专有微调，量化快速执行** | **训练侧**：基于自有小模型（如 Qwen2.5-7B 等基座）进行专项微调。训练使用实验室构造的独家"故障数据-专家诊断"真实数据对，**数据集严格保密，绝不公开**。<br>**部署侧**：通过 llama.cpp 进行量化压缩（4-5GB），实现单卡本地**快速推理执行**，对外暴露 OpenAI 兼容 API。 | **数据不出境**：私有化部署满足金融/政务等对数据隐私的绝对合规要求；SaaS 模式则仅上传脱敏后的结构化数字指标。<br>**长期壁垒**：闭源的独家数据集是核心护城河，"专属工具采集 -> 专家标注 -> 模型微调"的数据飞轮让竞争对手极难在短时间内逾越。 |
| **四、生态适配**<br>*(ARA 适配层)* | **标准协议驱动，沙箱隔离，授权穿透** | ARA 层基于 OAP 协议实现，各框架适配器以独立进程沙箱运行，统一 JSON Schema + OpenAPI 工具描述，License 授权粒度穿透至单框架/单场景级别。 | **防能力外泄**：授权穿透机制确保高级诊断工具不因外部框架集成而绕过 License 管控。<br>**生态护城河**：率先完成 openclaw / hermes / jiuwenclaw 深度适配，形成先发的生态绑定优势，后进者需重新完成适配投入。 |

### 三者之间的依赖与时序关系

这四块核心实现存在明确的依赖与时序要求，不可颠倒：

1. **Tool Runner 接口最先完成**：作为二进制工具接入与获取结构化输入的前提，接口设计直接决定后续开发顺利与否。
2. **二进制工具包早于模型上线**：高质量的工具输出数据是模型训练的核心语料。先有工具采集结构化数据，才能训练优质模型，避免数据格式不一致。
3. **模型最后交付但需长期投入**：随着工具覆盖场景增加，模型需持续利用新增数据进行增量微调，持续转动数据飞轮。
4. **ARA 适配层与工具包并行推进**：ARA 层依赖 Tool Runner 接口规范（已在步骤 1 完成），可与工具包开发并行推进，各框架适配器按优先级排期交付，不阻塞核心诊断链路上线。
