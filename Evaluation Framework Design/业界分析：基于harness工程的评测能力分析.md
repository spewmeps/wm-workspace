## 一、什么是 Harness 工程

**Agent Harness** 是围绕 AI 模型构建的软件基础设施，负责管理模型本身无法处理的一切——工具、记忆、上下文、安全检查、错误恢复，以及任务的完整生命周期。最简洁的公式：

> **Agent = Model + Harness**

这一概念由 Mitchell Hashimoto 提炼自 OpenAI 内部工程实践，随后经由 [Martin Fowler / Birgitta Böckeler 的系统化文章](https://martinfowler.com/articles/harness-engineering.html)（2026 年 4 月）成为行业标准词汇。

Fowler 将 Harness 组件分为两类：

- **Guides（前馈控制）**：在 Agent 行动前约束和引导它，如系统提示、规则文档、架构约束。
- **Sensors（反馈控制）**：在 Agent 行动后观察和验证其行为，是评测机制的核心载体。

Harness 工程与此前工程范式的区别在于：它不是 Prompt 工程（后者只写单轮指令），不是模型选型（模型越来越商品化），也不只是编排框架（LangChain 等处理的是编排，Harness 工程涵盖完整的控制系统）。Fowler 本人在另一篇文章 [Humans and Agents in Software Engineering Loops](https://martinfowler.com/articles/exploring-gen-ai/humans-and-agents.html) 中也指出：构建和维护 Harness，才是人类在高吞吐 Agent 时代唯一可规模化的参与方式。

## 二、评测在 Harness 工程中的核心地位

在 Harness Engineering 的语境里，评测不是一个附属能力，而是整个系统可控、可迭代、可规模化的前提。Fowler 更常使用 **Sensors** 这一术语，本质上讨论的正是 Agent 系统中的评测机制——它不是事后统计分数的报表模块，而是**运行时持续感知质量、驱动纠偏和推动系统演进的反馈层**。

### 2.1 让 Harness 从"提示词集合"升级为闭环控制系统

Harness 的核心是围绕 Agent 构建一个可被约束、可被观察、可被修正的控制系统。Guide 负责前馈控制，告诉 Agent"应该怎么做"；评测对应的 Sensor 负责反馈控制，回答 Agent"刚才做得对不对"。

- 只有 Guide，没有评测：Agent 只能被动执行指令，却无法验证规则是否真正生效。
- 只有评测，没有 Guide：系统在一次次报错中反复试错，缺乏稳定的行为约束。

因此，**评测的首要价值在于它让 Harness 形成闭环**。没有评测，Harness 只能提供约束，不能形成反馈；不能形成反馈，就无法称之为真正的工程系统。

### 2.2 决定 Agent 能否实现自我修正

传统软件中，测试发生在执行之后；Harness 工程里，高价值评测更强调**运行中的即时反馈**。一个设计良好的 Sensor，不只是指出错误，还要输出可被 LLM 直接消费的修正信号，例如：

- 哪个步骤失败了
- 失败原因是什么
- 建议下一步如何修复
- 是否应该重试、降级或请求人工介入

这类评测信号本质上是一种面向 Agent 的结构化反馈，直接决定 Agent 是否能从"执行任务"进化为"发现问题并自我修正"。因此，评测不只是"验收结果"，而是**参与生成过程本身**。

### 2.3 质量左移与分层治理的基础设施

Harness 工程面对的是一个跨生命周期的质量体系问题，评测需要被分布到不同阶段，以不同成本和不同速度介入：

| 阶段 | 评测作用 | 典型形式 |
|------|----------|----------|
| Agent 执行中 | 发现错误并触发即时修正 | 格式校验、工具调用检查、循环检测、策略偏航检测 |
| 提交前或自动化流程中 | 在低成本阶段尽早拦截问题 | lint、类型检查、规则校验、结构化 score |
| 人工审查和 CI 中 | 对复杂语义与业务目标进行判断 | LLM-as-a-Judge、人工 review、回归对比 |
| 长周期运行中 | 监控系统是否发生漂移和退化 | 成功率趋势、异常日志分析、版本对比、实验评测 |

**谁来发现问题、何时发现问题、发现后如何进入下一个动作**，这些都依赖评测设计来完成。评测是 Harness 质量体系的承载方式，而非某一个打分器。

### 2.4 Harness 持续演进的驱动信号

Harness 工程的目标不是一次性把规则设计完，而是在真实运行中不断校正 Guide 和 Sensor。评测结果承担的是一种"系统诊断信号"的角色：

- 某类错误反复出现 → Guide 不足，需要补充前馈约束
- 某类问题经常漏检 → Sensor 不足，需要加强反馈覆盖
- 某个版本分数持续下降 → Harness 改动可能引入了退化
- 某些分支收敛效果不稳定 → 协作策略或评测方法本身需要重构

评测不仅用于判断 Agent 输出是否合格，更用于判断 **Harness 自己是否仍然有效**。没有评测数据，人类无法知道系统到底该优化哪里。

### 2.5 最难标准化，也因此最有竞争价值

静态检查、格式校验等 Computational Sensor 相对成熟，但一旦进入以下问题，通用平台能力就会迅速不足：

- **工具调用路径是否符合预期**

  Agent 完成一个任务，背后往往要按顺序调用十几个工具：先查数据库、再调用外部 API、再写文件、再通知下游服务。这个调用顺序和组合方式，是由业务逻辑决定的，走错了哪一步都可能导致连锁问题。
  
  问题在于：通用平台只能告诉你"这个工具被调用了"，却无法判断"这个工具是不是在对的时机、以对的参数被调用的"。好比你让一个实习生去完成一笔采购，你不只想知道最后东西买没买到，还想知道他有没有先审批、再询价、再下单——跳过了审批直接下单，即便这次侥幸没出问题，下次也会踩坑。
  
  要验证"调用路径是否符合预期"，评测系统必须事先知道你的业务规定的正确路径长什么样，并且能把实际执行路径和预期路径做逐步对比。这套逻辑没有任何平台能帮你内置，只能在 Harness 的 Sensor 层自己实现，因为每个系统的"正确路径"定义都不一样。

- **多 Agent 并行过程是否正确收敛**

  复杂任务常常由多个 Agent 分头处理：一个负责搜集信息、一个负责分析、一个负责生成方案、一个负责做最终决策。这些 Agent 并行跑完之后，要把各自的结果汇总成一个统一的输出。
  
  这里的问题不是"每个 Agent 有没有跑完"，而是"汇总之后的结论有没有逻辑一致性"。就像四个人同时去调研一个市场，最后要出一份报告——如果 A 说市场规模是 100 亿，B 说是 30 亿，最后报告里两个数字都出现了，那这个任务就没有真正收敛，尽管四个人都"完成"了自己的那部分。
  
  判断收敛质量，需要理解任务的整体目标，以及不同分支的结果之间应该满足什么约束关系。这种判断标准是业务特有的，不同任务之间几乎没有通用性可言，评测逻辑必须由 Harness 自己来定义和执行。

- **记忆是否在正确轮次被调用**

  支持多轮对话的 Agent 会积累"记忆"：用户之前说过的偏好、已经确认过的决策、中间产生的中间结果等等。但记忆的价值不只在于"有没有存下来"，更在于"在该用的时候用了没有，在不该用的时候有没有被错误地带入"。
  
  举个具体场景：用户在第一轮告诉 Agent"我们公司不接受超过 30 天的账期"，Agent 记住了。但在第五轮生成合同草稿时，Agent 忘记调用这条记忆，草稿里账期写成了 60 天。表面上看，Agent 每一步都有输出，记忆也确实存在系统里，但这次评测应该判定为失败——因为关键约束在关键时刻没有被正确应用。
  
  反过来，也可能出现"记忆污染"：某个历史上下文不应该影响当前任务，但 Agent 错误地把它带了进来。要检测这两类问题，评测系统必须知道"哪条记忆应该在哪一轮发挥作用"，这是纯粹的业务知识，任何通用平台都无法预置。

- **某个 sprint contract 是否真正达成**

  在 Harness 工程里，sprint contract 是每个迭代开始时约定的一组交付标准，比如"完成用户权限模块的重构，覆盖率不低于 85%，所有已有接口保持向后兼容，并通过性能基线测试"。这不是一个单一的布尔值，而是一组有层次的验收条件。
  
  判断 contract 是否"真正达成"，远比"代码有没有提交"复杂得多。代码提交了，但兼容性测试没过，算达成吗？覆盖率 84.9%，算达成吗？性能测试通过了，但只测了主流程没有测边界情况，算达成吗？这些判断完全取决于团队在 contract 里怎么定义"达成"的边界，不同团队、不同项目之间的标准差异极大。
  
  更难的是，有些 contract 里的条件是定性的而不是定量的，比如"架构不引入新的循环依赖"或"新功能与现有设计风格保持一致"。这类条件需要结合代码结构、历史设计决策等多维上下文才能判断，是高度定制化的评测逻辑，通用平台根本无从内置。

- **行为正确性是否能被稳定评估**

  前面四个问题是"评测什么"的难题，这个问题则是"如何评测"本身的难题，而且往往更隐蔽、更难处理。
  
  很多任务的正确性判断标准在本质上就是模糊的。比如让 Agent 生成一段客服回复，要求是"专业但不冷漠，简洁但不失温度"——这个标准如果交给三个人来打分，可能会得出三种不同的结论。如果用 LLM 来打分，同样的输出在不同的评测 prompt 下也可能得到截然不同的分数。
  
  更棘手的是，即便你今天把评测标准调校得很稳定，随着模型版本升级、业务规则调整、上下文样本分布变化，评测结果本身也可能发生漂移——不是 Agent 变差了，而是评测系统对"正确"的判断悄悄变了。
  
  这意味着评测本身也需要被持续校准和维护，它不是一个配置一次就能永久信任的模块。这种"评测的可靠性"问题，只有在具体业务场景下，通过大量真实样本反复标注和对比，才能逐渐建立起来。这是一项需要长期投入的系统性工程，也是为什么说评测能力是 Harness 工程核心竞争力的根本原因。

这些问题都带有强烈的业务语义和系统上下文依赖，决定了评测逻辑必须深度嵌入 Harness 本身，而不能完全外包给通用观测平台。**评测能力本身，就是 Harness 工程竞争力的一部分。**

> 小结：如果说 Guide 定义了 Agent"应该做什么"，那么评测定义的就是系统"如何确认它真的做对了，以及做错后怎么办"。前者决定上限，后者决定可靠性；在真实工程里，决定系统是否可上线、可维护、可扩展的，往往正是后者。

## 三、Langfuse 能力对比：逐维评估

以下从 Harness 工程的评测诉求出发，逐维评估 Langfuse 当前能力的满足程度。

| 评测维度 | Harness 工程的诉求 | Langfuse 当前能力 | 满足度 | 核心限制 |
|----------|--------------------|-------------------|--------|----------|
| **多步骤轨迹（Trajectory）评测** | 记录每个 step 的决策路径、输入输出，评估整体轨迹是否合理 | Trace + Observation 层级模型可记录每个 step；支持 trace-level 和 observation-level 打分 | 🟡 基本满足 | 轨迹的 DAG/图状可视化不成熟；并行分支的轨迹展示仍为树状，复杂 harness 结构难以直观呈现 |
| **工具调用（Tool Call）正确性验证** | 验证 Agent 是否在正确时机调用了正确工具，参数是否符合预期 | 可记录每次 tool call 的 observation；支持将验证结果以 score 写入；2026 年已强化 tool call 渲染 | 🟡 部分满足 | **不内置** "预期调用路径 vs 实际调用路径" 的 diff 逻辑；正确性判断须在 harness sensor 层自行实现后写入 Langfuse |
| **任务完成率 / Success Rate** | 统计每次 harness 运行的整体任务是否成功完成 | Boolean/Numeric score 原生支持；Dashboard 可聚合完成率，支持跨版本趋势对比 | 🟢 完全满足 | 无明显限制，开箱即用 |
| **中间步骤打分（Step-wise Eval）** | 对 harness 每个执行节点（需求解析、skill 调用、记忆更新等）独立打分 | Observation-level LLM-as-a-Judge（2026.02 重点强化），秒级异步评测；自定义 score 可写入任意 observation | 🟢 满足 | LLM-as-Judge 内置模板无法覆盖 harness 特有的业务语义（如"sprint contract 是否达成"），需要自定义评测 prompt |
| **多轮对话质量评估** | 评测跨轮次的上下文连贯性、记忆利用准确率、目标一致性 | Session 模型支持多轮归组；session-level score 可挂载 | 🟡 部分满足 | 记忆利用率、上下文连贯性等语义评测**无内置支持**；harness 的记忆模块验证逻辑须自建 |
| **并行协作分支评测** | 多 Agent 并行执行时，评测各分支结果是否正确收敛 | 可通过多 trace 关联或 session 归组记录并行执行结果 | 🔴 基本不满足 | 无原生的多 Agent 并行轨迹合并视图；分支收敛质量评估完全依赖外部逻辑 |
| **Sensor 反馈信号采集** | 将 harness 的 sensor 输出（格式校验、安全检查、循环检测等）结构化记录 | 任意自定义 score 均可通过 SDK 写入，支持 Numeric/Boolean/Categorical/Text 四种类型 | 🟢 满足 | sensor 类型丰富时，score 命名和分类管理需要团队自行约定规范 |
| **回归检测（版本对比）** | 新版 harness 上线前，验证是否引入评测分数退化 | Experiments 实验对比、dataset run 版本对比；2026.04 Experiments 已成为一等公民 | 🟢 满足 | 对 harness 特有的"并行多 Agent 版本对比"场景，实验对比 UI 的粒度还不够细 |

## 四、综合差距分析：Langfuse 的四大缺口

**整体定位差距**

Langfuse 是**评测结果的基础设施**（存储、聚合、可视化、趋势追踪），Harness 工程是**评测逻辑的执行者**（运行 Agent、触发 sensor、产出 score）。两者天然互补，但 Langfuse 无法替代 harness 内部的验证逻辑。按照当前能力估算，Langfuse 能满足 Harness 工程约 **60%** 的评测基础设施需求，其余 40% 集中在以下四个方向：

> **📌 贯穿全节的示例场景**
>
> 假设你构建了一个**代码审查 Agent 系统**：用户提交一个 Pull Request，系统启动后由三个 Agent 并行工作——A 负责分析代码规范合规性，B 负责评估性能影响，C 负责检查安全漏洞——最终由一个 Coordinator Agent 汇总三份报告，生成审查结论，并记住该项目历史上曾经豁免过哪些规则。如果结论不符合团队事先约定的 sprint contract（如"所有 P0 安全漏洞必须标记为阻断合并"），系统应立即触发重试并通知负责人。
>
> 以下四个缺口，都将在这个场景中具体呈现。

**缺口一：缺乏 Harness 业务语义的原生支持**

Langfuse 的评测模板（toxicity、relevance、correctness 等）面向通用 LLM 应用，对 harness 特有概念（sprint contract、skill 路由决策、guide vs sensor 结果）没有内置映射，需要大量自定义评测 prompt 来弥补。

在上述场景中，"所有 P0 安全漏洞必须标记为阻断合并"是团队自己定义的 sprint contract，Langfuse 的内置模板里完全没有这个概念。你无法直接配置一条规则说"如果 Agent C 发现了 P0 漏洞但最终结论里没有出现阻断标记，则评测失败"。你必须自己写一套评测 prompt，教会 LLM-as-a-Judge 去理解什么是 P0、什么是阻断、两者之间的业务关联是什么——而这套逻辑对于每个团队、每个项目都可能不同，Langfuse 永远无法帮你预置。

**缺口二：并行多 Agent 轨迹无法结构化呈现**

Harness 工程普遍涉及 Planner/Generator/Evaluator 等多 Agent 协作，Langfuse 目前只有单线程树状 trace 视图，复杂并行结构的可视化和归因分析能力明显不足。这对于理解分支收敛失败的根因尤为不利。

在上述场景中，Agent A、B、C 是并行运行的，最终由 Coordinator 汇总。假设某次审查结论出现了遗漏：P0 漏洞没有被标记为阻断。你打开 Langfuse 想排查原因——是 C 没有识别出这是 P0？还是 C 识别了但 Coordinator 汇总时丢失了这条信息？还是 Coordinator 汇总正确，但最终生成文本时格式错误导致标记丢失？Langfuse 的树状 trace 视图会把 A、B、C 的执行记录呈现为三条独立的链路，你很难在一个统一视图里看清三者的时序关系、中间结果如何流入 Coordinator，以及哪一步是真正的失效节点。归因分析的大量工作只能靠人工翻日志来完成。

**缺口三：记忆模块评测完全空白**

Harness 的记忆机制（跨轮次信息提取与利用）是其核心竞争力之一，而"记忆是否被正确调用"类指标在 Langfuse 中没有任何内置支持，需要完全在 harness sensor 层自建并以自定义 score 形式写入。

在上述场景中，系统会记住该项目历史上曾经豁免过某些规则——比如这个项目曾经团队决议豁免了某类第三方库的版本锁定检查。当新的 PR 触发审查时，Agent A 应该自动调取这条记忆，跳过对该规则的检查。如果它没有调取，就会产生误报；如果它在另一个不相关的项目里错误地调取了这条记忆，就会产生漏报。这两种情况 Langfuse 都无法自动检测——它只知道"这一轮 Agent A 输出了什么"，不知道"它本该从记忆里读取什么、实际上读取了没有"。你必须在 Harness 的 Sensor 层自己实现一套记忆核查逻辑，记录预期应调取的记忆 ID 和实际调取的记忆 ID，做 diff 后把结果以自定义 score 写入 Langfuse，它才能统计和追踪这个指标。

**缺口四：实时 Sensor 反馈闭环缺失**

Harness 中 sensor 检测到问题后通常需要触发 retry 或 escalation，而 Langfuse 目前是被动接收评测结果，不支持主动触发 harness 侧的响应动作（webhook 触发等能力还在早期阶段），实时闭环反馈尚未形成。

在上述场景中，假设 Sensor 检测到 Coordinator 的最终结论里缺少了对 P0 漏洞的阻断标记，按照设计，系统应该立即触发重试——把三个 Agent 的原始报告重新喂给 Coordinator，并附上修正提示"请确认是否遗漏了 P0 安全问题的阻断标记"。但 Langfuse 只能被动地把这次 sensor 检测的结果记录下来，它无法主动告诉 Harness"你现在需要重试"。触发重试的逻辑必须完全写在 Harness 自身的 Sensor 层里，Langfuse 对这个闭环动作没有任何感知和参与。这意味着，你在 Langfuse 里看到的永远是"已发生的结果"，而不是"系统正在自我修正的过程"，实时闭环对它来说是一个黑盒。

---

**结论**：Langfuse 能很好地承担 score 存储、任务完成率统计、step-wise 打分、回归检测等基础能力，但在 Harness 特有的并行协作评测、业务语义自定义、记忆模块评测、实时闭环反馈四个方向上存在明显缺口，需要在 harness sensor 层自建补齐，或结合专门的 Agent 评测框架（如 AgentBench、τ-bench）联合使用。

## 五、项目案例：`witty-skill-insight` 的增强方向

基于目前可获取的公开生态信息、项目命名以及 openEuler Witty Assistant 的整体上下文，可以将 `witty-skill-insight` 理解为 Witty Assistant 生态中的一个开源评测/洞察组件。结合名称中的 `skill-insight` 语义和运维场景特征，它更像是一个围绕 Agent Skill 执行效果进行观察和分析的轻量评测层，其关注点大概率集中在两个方面：

- **历史案例匹配质量**：知识库召回是否相关、是否有助于问题定位
- **日志异常检测结果的可读性**：异常提取是否清晰、最终分析报告是否便于人理解和消费

如果将这个项目放回前文所述的 Harness 工程框架中看，它当前更接近一个**轻量 Sensor**: 能感知输出质量，但还没有完全进化为一个**深度参与执行闭环、能够驱动修正动作的反馈层**。从 Harness 工程对评测的完整要求出发，`witty-skill-insight` 至少存在以下五个值得优先考虑的增强方向。

以下是这五个增强方向的总体概览：

| 增强方向 | 当前状态与局限 | 核心增强策略 | 预期价值与意义 |
| --- | --- | --- | --- |
| **1. Skill 调用路径验证** | 侧重验证最终输出的可读性，对过程正确性缺乏检查 | 引入“预期调用链”机制，对实际路径与预期路径进行 diff | 确保执行时机、顺序与参数正确，提升系统的鲁棒性 |
| **2. 运维记忆正确性评测** | 仅关注召回数量，无法评估是否“漏召回”或“错应用” | 构建黄金样本集，校验目标案例的召回命中率与引用覆盖率 | 闭环验证，保障系统“历史经验复用”这一核心能力 |
| **3. 并行收敛质量评测** | 视角局限于单线程，无法评估多分支结果的融合质量 | 预留多路输出摘要、关键信号覆盖率及冲突检查的数据结构 | 为未来向多 Agent/多 Skill 复杂并行架构演进奠定基础 |
| **4. 驱动即时重试闭环** | 仅作为事后的质量报表，无法在运行中触发纠偏 | 输出包含执行建议的结构化反馈信号，驱动自动重试或转人工 | 推动组件从“质量展示层”真正升级为 Harness 的“反馈层” |
| **5. 跨版本回归检测** | 缺乏因知识库扩容或模型升级导致退化的自动检测手段 | 建立长期基线测试集，在发生关键变更时自动回跑对比 | 防范隐性退化（静默失效），保障系统长期运行的稳定性 |

接下来我们对这五个方向进行详细展开。

### 5.1 从“结果可读性”升级为“Skill 调用路径验证”

当前这类 insight 组件最容易做的是检查最终报告是否结构完整、根因描述是否清晰、结论是否可读。但在 Harness 工程视角下，更关键的问题其实是：**Agent 在生成这份报告的过程中，是否调用了正确的 Skill，以正确的顺序，传入了正确的参数。**

以“系统启动异常”的运维工单为例，合理的 Skill 调用链通常应当是：

1. 先调用日志异常检测 MCP，提取关键异常片段
2. 再调用知识库 MCP，进行历史案例匹配
3. 最后综合生成根因分析和处置建议

如果 Agent 跳过日志检测，直接使用原始日志做案例匹配，或者把错误的参数传给知识库检索，即便最终报告“看起来还能用”，底层路径仍然是脆弱的。一旦日志样本稍有变化，系统就可能失效。

**增强方向**：在 `witty-skill-insight` 中引入“预期 Skill 调用链”机制。为每类常见故障定义标准调用模板，在每次执行后把实际路径与预期路径做 diff，记录偏差类型，例如：

- 跳过关键步骤
- 参数异常
- 调用顺序错误
- 多余调用或重复调用

这样评测就不再只停留于“结果像不像对的”，而是能进一步回答“过程是否按正确方式完成”。

### 5.2 增加运维记忆的正确性评测

Witty Assistant 的核心价值之一，是把历史运维经验沉淀到知识库中，并在新故障出现时进行复用。从 Harness 工程视角看，这本质上就是一种**记忆调用机制**。但如果评测只能看到“召回了几条案例”，却看不到“该召回的案例有没有被召回、召回后有没有被正确使用”，那么这个系统最关键的能力其实仍然处于黑盒状态。

举例来说，如果某个“海思驱动导致系统挂死”的经典案例已经进入知识库，那么面对症状高度相似的新工单，Agent 应该优先召回这条案例并复用其中的解决路径。风险主要有两类：

- **漏召回**：该召回的案例没有被召回，导致分析偏离
- **错应用**：案例召回正确，但在最终报告里没有把关键经验真正应用出来

**增强方向**：建立“记忆召回准确率”的评测闭环。可以在运维案例库里构造一批已知答案样本，标注每类故障对应的目标案例 ID，然后在每次 Agent 执行后校验：

- 实际召回结果是否包含目标案例
- 目标案例是否进入后续推理和结论生成
- 召回质量在不同版本间是否退化

在指标层面，可以优先引入 `Recall@K`、目标案例命中率、案例引用覆盖率等轻量指标。这类改造的工程成本相对可控，但对提升系统可靠性非常直接。

### 5.3 预留多 Skill 并行协作的收敛质量评测

当前 Witty Assistant 可能仍以单 Agent 串行调用 Skill 为主，但随着运维场景复杂度提升，未来很可能演进为多 Skill 乃至多 Agent 并行协作。例如：

- 一路做日志异常检测
- 一路做历史案例检索
- 一路做系统配置核查
- 最后由汇总 Agent 输出统一结论

一旦进入这种模式，评测的重点就不再只是“每一路输出是否正确”，而是**最终结论是否正确吸收了各分支的关键信息，并完成一致性收敛**。

**增强方向**：即便当前还是串行执行，也建议在 `witty-skill-insight` 中先预留“多路结果收敛评测”的数据结构，例如：

- 每路 Skill 的输出摘要
- 每路结果中识别出的关键信号
- 最终报告对这些关键信号的覆盖率
- 分支结论之间是否存在冲突

这样做的价值在于，一旦未来架构升级为并行协作，评测体系不需要从零重建，而是能顺滑扩展。

### 5.4 让 Sensor 信号驱动即时重试闭环

如果 `witty-skill-insight` 只是一个“任务结束后打分和展示结果”的组件，那么它的价值更接近质量报表，而不是 Harness 工程里真正的 Sensor。后者的关键能力在于：**当检测到问题时，系统可以立即采取修正动作。**

例如，在一次故障分析中，Sensor 发现：

- 案例召回质量过低
- 根因定位缺少关键日志片段
- 最终结论遗漏了风险等级判断

理想状态下，系统不应该只是把这些问题记录下来供事后查看，而应该立刻触发下一步动作，比如：

- 自动扩大检索范围并重试
- 补充日志上下文后重新分析
- 超过重试阈值后转人工处理

**增强方向**：让 `witty-skill-insight` 输出的不只是“分数”，而是包含可执行建议的结构化反馈信号，例如：

- 哪一步失败了
- 失败原因是什么
- 建议采取何种重试策略
- 是否需要人工介入

一旦 Witty Assistant 的执行层开始消费这些结构化信号，`witty-skill-insight` 就会从“质量展示层”升级为“闭环反馈层”。

### 5.5 建立跨版本回归检测与 Skill 质量基线

对于依赖案例库积累和 Prompt/模型持续迭代的运维系统来说，最危险的问题往往不是“明显崩溃”，而是**悄悄退化**。例如：

- 案例库扩容后，旧问题的匹配准确率反而下降
- Skill Prompt 调整后，报告可读性提高了，但根因判断准确率变差
- LLM 版本升级后，日志摘要更流畅了，但关键异常被遗漏

如果没有回归检测，这类问题很难被及时发现。

**增强方向**：从现有运维案例库中抽取一批“黄金样本”作为长期基线测试集，形成稳定的回归评测套件。每次发生以下变更时自动回跑：

- 案例库更新
- Skill Prompt 调整
- MCP 检索逻辑变更
- 底层模型升级

对比的核心不只是总分，还包括：

- 案例召回准确率
- 根因定位准确率
- 报告完整性
- 关键风险信息覆盖率

这本质上就是把 Harness 工程中的“回归检测”能力，落到运维 Agent 的具体场景中。

### 小结

从当前形态看，`witty-skill-insight` 已经是一个有价值的起点，它至少明确认识到了“Agent 输出质量需要被观察”这件事。但若放到 Harness 工程的完整框架下，它目前更像是一个**输出质量的展示层**，而不是一个**参与执行过程、能够驱动系统修正的反馈层**。

按优先级来看，更建议优先推进以下两项：

1. **记忆召回准确率评测**：改造成本较低，对结果正确性提升直接
2. **Sensor 信号结构化**：一旦落地，就能为自动重试和人工兜底建立闭环基础

在此基础上，再逐步扩展到 Skill 调用路径验证、并行收敛评测和跨版本回归检测，`witty-skill-insight` 才有机会从“事后洞察组件”演进为 Harness 工程意义上的真正 Sensor。
