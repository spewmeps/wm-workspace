## 1. OpenClaw 架构全景

OpenClaw 采用 **调度中心架构 (Dispatch Center Architecture)**。所有消息都经过一个中央塔台——**Gateway 网关**，由它分配到正确的通道（Agent）。其核心设计思想是将**消息通信**、**接口层**与**AI 决策执行 (Agent)** 彻底解耦。

```text
┌─────────────────────────────────────────────────────────────────┐
│                        CHANNEL  通信渠道                         │
│                                                                  │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐  ┌────────────────┐  │
│  │ WhatsApp │  │ Telegram │  │ Discord  │  │    iMessage    │  │
│  └──────────┘  └──────────┘  └──────────┘  └────────────────┘  │
│                      + 插件扩展 Plugins                          │
└───────────────────────────┬─────────────────────────────────────┘
                            │
                    消息 / 指令 ingress
                            │
                            ▼
┌─────────────────────────────────────────────────────────────────┐
│                      GATEWAY  网关服务                           │
│                       （统一入口）                               │
│                                                                  │
│   WebSocket  ws://127.0.0.1:18789        ← loopback only        │
│   HTTP       http://<gateway-host>:18793                        │
│              └── /__openclaw__/canvas/   ← Canvas 主机          │
└───────────────────────────┬─────────────────────────────────────┘

     │  AGENT     │  │   NODE       │  │   NODE            │
     │  RUNNER    │  │   设备       │  │   设备             │
     │            │  │              │  │                   │
     │ 🤖 Pi智能体│  │ 💻 CLI指令   │  │ 📱 iOS 节点       │
     │   (RPC)    │  │   (电脑)     │  │   (GW WS + 配对)  │
     │            │  │              │  │                   │
     │💬 聊天 UI  │  │ 🍎 OpenClaw  │  │ 🤖 Android 节点   │
     │ (SwiftUI)  │  │   .app(macOS)│  │   (GW WS + 配对)  │
     └────────────┘  └──────────────┘  └───────────────────┘
```

消息进入 Gateway 后，系统统一执行以下任务：
1.  **渠道协议适配**：屏蔽不同 IM 平台的协议差异。
2.  **会话与上下文管理**：加载历史记录与工作区配置。
3.  **RPC 调度**：将任务分发至 Pi 智能体。
4.  **响应回传**：将结果流式返回给用户。

**核心模块：**
*   **Gateway (网关)**：系统的单一事实源 (Single Source of Truth)，集中处理认证、流控与路由。
*   **Agent (智能体)**：执行引擎，负责组装上下文、调用模型、执行工具操作。

**边界说明：**

| 组件 | 所属层 | 依据 |
|------|--------|------|
| Ingress / Channel Adapter | Gateway | 纯传输层，格式标准化 |
| Routing / Session Router / Lane Queue | Gateway | 流量控制，不含推理 |
| **Context Assembly** | **Agent Runtime** ✅ | Agent Runner 负责 prompt 组装 |
| **LLM Inference** | **Agent Runtime** ✅ | Agent Runner 调用外部 LLM |
| **Tool Execution（Agentic Loop）** | **Agent Runtime** ✅ | 源码 `pi-embedded-runner` 中实现 |
| Streaming + Persistence | Agent Runtime | 回复成形与落盘在 agent 侧完成 |
| Egress 回传路由 | Gateway | 最终由 Gateway 发回渠道 |

#### 2.1.2 协议与关卡
*   **协议分工**：HTTP 处理静态资源与工具端点；WebSocket 处理实时通信与核心控制。
*   **四道安全关卡**：
    1.  **Access Control**：决定谁有资格发送消息。
    2.  **Agent Routing**：精准匹配目标 Agent。
    3.  **Session Resolution**：通过 SessionKey (`Agent+平台+类型+用户ID`) 锁定上下文。
    4.  **Runtime Dispatch**：移交 Agent Runtime 执行。

#### 2.1.3 异步架构设计
*   **Event-Driven**：全异步流转，高并发互不阻塞。
*   **Smart Streaming**：分块平滑输出，优化用户体验。
*   **Lane Queue**：默认串行 (Session 级锁)，显式并行 (Cron 任务)，保障数据一致性。

### 2.2 通信渠道 (Channels)

OpenClaw 将每个 IM 实例视为一个 **Channel**，包含两大组件：
*   **Monitor (听)**：监听 Webhook/长轮询，校验权限，归一化消息格式。
*   **Adapter (说)**：将内部格式转为平台 API，处理分块、渲染及重试。

### 2.3 节点设备 (Node)

**Node** 是运行在终端设备 (iOS/Android/macOS) 上的轻量级程序，赋予 OpenClaw 对物理世界的感知与控制能力。

*   **发现机制**：局域网内通过 **mDNS/Bonjour** 自动发现；广域网通过 **Tailnet** (Tailscale) 组网。
*   **安全通信**：基于 Tailnet 构建虚拟局域网，无需暴露公网端口即可实现端到端加密控制。

### 2.4 Agent Runtime & Core：智能体内核

#### 2.4.1 核心哲学
OpenClaw 摒弃复杂的规划器，采用 **极简主义** 与 **文件驱动** 哲学。

> **核心判断**：传统框架常因同步阻塞导致长任务失败。OpenClaw 基于 **事件驱动的 π (pi) 运行底座**，默认开启 **YOLO 模式** (完全自主执行)，支持真正的后台异步自动化。

| 对比维度 | Claude Code | Codex | Pi (OpenClaw) |
| :--- | :--- | :--- | :--- |
| **系统提示词** | ~10k tokens | 适中 | **< 1000 tokens** |
| **内置工具** | 数十个 | 适中 | **4个** (read/write/edit/bash) |
| **Plan Mode** | 黑盒 | 有 | **无** (文件驱动) |
| **架构哲学** | 编排型 | - | **自主型** (Agent 决定) |

#### 2.4.2 内核架构 (pi-agent-core)
采用严格分层的 Monorepo 架构：
1.  **Foundation (pi-ai)**：统一模型抽象，Schema 校验，流式协议处理。
2.  **Core (pi-agent-core)**：
    *   **Event Bus**：系统流转图腾，所有操作皆事件。
    *   **StateMachine**：INIT -> RUNNING -> SUSPENDED -> FINISHED。
    *   **Executor**：基于 Tick 的心跳驱动机制。
3.  **Applications**：具体的 Agent 实现 (如 Coding Agent)。

#### 2.4.3 上下文与工具
*   **Context**：动态组装 System Prompt + Bootstrap Context (Markdown 文件) + Session History。
*   **Tools**：仅暴露 `read`, `write`, `edit`, `bash` 四大原语，迫使模型像人类程序员一样思考。

### 2.5 记忆系统 (Memory System)

采用 **"文件即记忆"** 设计，所有记忆存储为 Markdown 文件：
*   **静态配置**：`IDENTITY.md` (身份), `SOUL.md` (准则), `AGENTS.md` (能力)。
*   **动态记忆**：`memory/YYYY-MM-DD.md` (短期), `MEMORY.md` (长期)。

#### 跨会话记忆 (RAG)
监听 `memory` 文件夹变化，触发 Embedding 并写入 SQLite `chunks` 表，支持混合搜索 (RAG + BM2.5)。

```sql
CREATE TABLE chunks (
  id TEXT PRIMARY KEY,
  path TEXT, source TEXT,
  start_line INTEGER, end_line INTEGER,
  hash TEXT, model TEXT,
  text TEXT, embedding TEXT,
  updated_at INTEGER
);
```

### 2.6 多 Agent 协作 (Multi-Agent)

OpenClaw 支持在单 Gateway 实例中运行多个隔离的 Agent "大脑"。

1.  **架构原理**
    *   **全隔离设计**：每个 Agent 拥有独立的 Workspace、State Directory 和 Session Store。
        *   *Workspace 详解*：通过 `SOUL.md` (准则), `IDENTITY.md` (风格), `USER.md` (画像) 定义个性化人格。
    *   **沙箱隔离**：支持 Docker 容器化运行不可信 Agent。

2.  **实现流程**
    通过 `bindings` 配置确定性路由 (Channel/Account/Peer/Guild 维度)。支持 **Sub-Agents** (子智能体) 与 **ACP** (外部运行时) 扩展。

## 3. 总结：重新定义 Agent 工程化标准

OpenClaw 不仅仅是一个聊天机器人框架，更是一个为生产环境设计的 **Agent 操作系统**。

### 核心价值
1.  **调度中心架构**：Gateway 作为单一事实源，解耦通信与智能，保障高并发稳定性。
2.  **极简哲学**：回归 **LLM + 文件系统** 本质。**文件即记忆**，**工具即原语**，配合 **YOLO 模式** 实现真正的自主进化。
3.  **平滑演进**：原生支持从单体助手到 Multi-Agent 集群的扩展。
4.  **虚实结合**：通过 Node 与 Tailnet 连接物理世界与数字世界。

OpenClaw 证明了构建强大的 Agent 不需要堆砌复杂概念。通过**稳定的基础设施**、**透明的文件上下文**及**安全的执行环境**，我们能够构建出可靠、可控且无限扩展的智能体系统。它是目前探索 **Agent-Native** 应用架构的最佳范本之一。
